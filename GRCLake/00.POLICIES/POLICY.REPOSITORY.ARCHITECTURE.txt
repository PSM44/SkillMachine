==========
00.10_OBJETIVO
==========

Definir una política mandatoria para que el repositorio sea:

- entendible por humanos e IA (sin inventar)
- estable y escalable (sin drift)
- gobernable (cambios controlados)
- validable (quality gates automáticos)
- auditable (evidencia verificable)

Esta política es GRC, no “sugerencia”. Si falla: se bloquea avance (STOP THE LINE).


==========
00.20_JERARQUIA_CANONICA
==========

Regla dura de autoridad:

1) HUMAN
   - manda eternamente
   - canon y espíritu del sistema
   - cambios canónicos solo por humano vía HUMAN

2) GRC
   - enforcement operativo (policy/control/evidence/validation)
   - deriva de HUMAN para obligar cumplimiento

3) SKILLS
   - capabilities TI (how-to)
   - implementan lo exigido por HUMAN/GRC

4) USECASES
   - packages derivados (deploy/runbook)
   - no son fuente canónica

5) TOOLS/ARTEFACTOS/LOGS
   - capa operativa mutable


==========
00.30_SCOPE_Y_FUERA_DE_SCOPE
==========

SCOPE (cubre):
- arquitectura de carpetas root del repo
- canonical sources vs operational copies
- reglas mínimas de ubicación de artefactos
- reglas de “no mover sin CIS”
- reglas de evidencia y validación
- reglas para módulos de IA desktop (ingest sin duplicar canon)
- reglas para USECASE packaging

FUERA DE SCOPE (no cubre):
- contenido profundo de cada skill/policy/control (salvo estructura y ubicación)
- diseño funcional de herramientas externas
- refactor masivo automático de todo el repo sin CIS

03.00_CANONICALITY_Y_SINGLE_SOURCE_OF_TRUTH
==========

03.10_CANONICAL_SOURCES

Son canónicos (fuente de verdad):
- HUMAN\  (canon máximo)
- GRCLake\ (policies/controls/risks/evidence/registries GRC)
- SkillsLake\ (capabilities/habilidades; no enforcement)

No son canónicos:
- 90.USECASE\ (outputs derivados)
- SyS\A_Tools\Radar\ (snapshots)
- SyS\A_Tools\OneShots\Logs\ (logs)

03.20_COPIAS_OPERATIVAS

Regla:
- Un USECASE puede tener COPIA operativa de políticas/plantillas.
- La copia NO es canon.
- La copia debe declarar (mínimo) SOURCE, VERSION y COPY_DATE.

Requisito mínimo para copias dentro de 90.USECASE\<USECASE>\:
- SOURCE_CANONICAL_PATH
- SOURCE_VERSION
- COPY_DATE
- COPY_MODE (STATIC_USECASE_COPY)

Si no existe ese header: WARN (hoy), y debe evolucionar a FAIL en release posterior cuando el proceso esté maduro.


==========
04.00_CHANGE_CONTROL_(CIS)_Y_ROLLBACK
==========

04.10_REGLA_DURA_DE_CAMBIO

Prohibido:
- mover/renombrar/eliminar en forma masiva sin plan CIS
- hard delete por defecto

Permitido:
- cambios con CIS + evidencia + rollback

04.20_CIS_MINIMO

Un cambio estructural debe tener:
- Change Request (CR) con ID
- migration map (origen -> destino)
- motivo (rationale)
- riesgo + mitigación
- rollback plan
- evidencia post-implementación

04.30_SOFT_DELETE

Default para deletes:
- mover a QUARANTINE (no hard delete)
- mantener metadata (timestamp, motivo, CR, actor)


==========
05.00_GRC_ENFORCEMENT_REQUIREMENTS
==========

Para que un artefacto sea “GRC”, debe cumplir el estándar GRC-MVP:

MANDATORY_GRC_BLOCKS:
- POLICY
- CONTROL
- EVIDENCE MODEL
- VALIDATION
- SEVERITY
- CHANGE CONTROL

Referencia:
- GRCLake\06.TEMPLATES\TEMPLATE.GRC.MVP.txt

Regla dura:
- Si falta un bloque mandatorio, no se considera GRC enforceable.

Nota práctica:
- Skills pueden nacer desde HUMAN.
- Skills ascienden a GRC solo si: “tiene sentido” + cumple GRC-MVP.

==========
11.00_RIESGOS_Y_MITIGACIONES
==========

RISK_01 — Drift doctrinario
- Impacto: el repo se vuelve inconsistente y la IA inventa reglas.
- Mitigación: jerarquía canónica + gates + “single source of truth”.

RISK_02 — Refactor masivo sin CIS
- Impacto: rotura de rutas, manifests, validadores y pérdida de historia.
- Mitigación: CIS + rollback + soft delete.

RISK_03 — Duplicación de HUMAN en módulos/usecases
- Impacto: conflicto de canon.
- Mitigación: modules apuntan a HUMAN; no lo duplican.

RISK_04 — “GRC” sin enforcement
- Impacto: falsa sensación de control.
- Mitigación: GRC-MVP mandatorio + validación automatizada.


==========
12.00_CHANGE_CONTROL_(DEFINITIVO)
==========

Autoridad:
- HUMAN changes: solo humano, editando HUMAN.
- GRC policy changes: propuesta + aprobación humana + evidencia + version bump.

Proceso mínimo:
1) Proponer Change Request (CR)
2) Evaluar impacto (scope/quality/config)
3) Aprobar (humano)
4) Implementar (scripts/patches)
5) Validar (quality gates)
6) Registrar evidencia
7) Actualizar versión y changelog

LOG recomendado:
- SyS\CIS\YYYYMMDD.CIS.<ID>.txt
