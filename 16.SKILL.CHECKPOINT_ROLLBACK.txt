00.20_CONTEXTO
==========

Ejecución frecuente sin rollback:
- es suicidio operativo.
Objetivo:
- rollback determinístico y auditable.

==========
01.00_PRINCIPIOS
==========

- Antes de cualquier APPLY:
  checkpoint obligatorio.
- Si FAIL:
  rollback automático salvo orden humana distinta.
- Rollback se valida (no se asume).

==========
02.00_MODO_GIT_(PREFERIDO)
==========

02.10_PRECHECK
- Repo debe estar inicializado.
- working tree limpio o se crea stash controlado.

02.20_CHECKPOINT
- Crear branch:
  ai/<PROJ_ID>/<TASK_ID>
- Crear commit checkpoint:
  "AI_ORCH CHECKPOINT <TASK_ID>"

02.30_APPLY
- Aplicar cambios.
- Commit final:
  "AI_ORCH APPLY <TASK_ID>"

==========
03.00_MODO_SNAPSHOT_(FALLBACK)
==========

03.10_CUANDO
- Git no disponible o repo no inicializado.

03.20_DONDE
- \03_ARTIFACTS\SNAPSHOTS\SNAP-YYYYMMDD-HHMM-SEQ\

03.30_CONTENIDO_MINIMO
- manifest.txt (lista archivos tocados)
- before\ (copias previas)
- after\ (copias posteriores opcional)
- hashes.txt

==========
04.00_PROTOCOLO_DE_ROLLBACK
==========

04.10_EN_FAIL
- Git: reset --hard al checkpoint commit
- Snapshot: restaurar before\ sobre paths originales

04.20_EN OK
- No rollback.
- Mantener checkpoint para auditoría.

04.30_OVERRIDE_HUMANO
- El humano puede pedir “no rollback”.
- Debe quedar log.

==========
05.00_VALIDACION_POST_ROLLBACK
==========

- Confirmar:
  - hashes vuelven al estado previo (o al commit)
  - proyecto compila/abre (smoke test)
  - no quedan archivos huérfanos fuera de quarantine

Si no pasa:
- INCIDENT mínimo MEDIA.

==========
06.00_EVIDENCIA_Y_LOGGING
==========

Registrar:
- checkpoint_id
- comando exacto de rollback
- resultado (OK/FAIL)
- paths afectados

==========
07.00_ANTIPATRONES
==========

- “rollback” sin evidencia.
- “rollback” parcial y declarar OK.
- borrar snapshots para “limpiar”.
