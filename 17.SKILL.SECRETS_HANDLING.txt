Tu caso real:
- ya tuviste API key hardcodeada.
Eso es INCIDENT por riesgo.

==========
01.00_PRINCIPIOS
==========

- Secret nunca en repo.
- Secret nunca en logs.
- Secret siempre via env var.
- Si se detecta secret en archivo:
  - FAIL + quarantine + TROUBLE.

==========
02.00_UBICACION_Y_ALMACENAMIENTO
==========

- Preferido: variables de entorno (User scope)
- Alternativa: archivo fuera del repo:
  C:\Dev\.secrets\<PROJ_ID>\secrets.env
  (Nunca dentro de PROJECT_ROOT)

==========
03.00_INYECCION_EN_RUNTIME
==========

- Orquestador lee env vars y las pasa al runtime.
- Executor no imprime env vars.

==========
04.00_REDACTION_EN_LOGS
==========

- Antes de escribir logs:
  - reemplazar patrones secretos por "***REDACTED***"
- Redactar también snippets de archivos .env y similares.

==========
05.00_DETECCION_(PATRONES)
==========

Patrones mínimos:
- "sk-" tokens largos
- "apiKey" seguido de strings largas
- PEM blocks (-----BEGIN ...)

Acción:
- si detecta => marcar evento WARN/ERROR y bloquear apply.

==========
06.00_INCIDENTES
==========

- Secret expuesto => INCIDENT severidad ALTA mínimo.
- Se requiere:
  - rotar key
  - limpiar historial si aplica
  - registrar postmortem breve
