20260416.1_IASKILLS.GIT.TXT
00.00 NOMBRE DEL SKILL

GIT.NAMING.CONVENTION.V1

01.00 OBJETIVO

Definir un estándar único, consistente y ejecutable para nombrar:

commits
branches
tags

Este estándar busca:

trazabilidad temporal exacta
legibilidad humana inmediata
compatibilidad con automatización IA
integración con BATON y RADAR
reducción de ambigüedad operativa
02.00 PRINCIPIOS DEL MODELO

02.01 TIEMPO COMO EJE PRINCIPAL
Todo elemento debe iniciar con timestamp:

Formato:
yyyymmdd.HHmm

Ejemplo:
20260416.1135

02.02 TIPO EXPLÍCITO
Siempre indicar el tipo de cambio:

Tipos permitidos:

INIT
FEAT
FIX
REFACTOR
DOC
AUDIT
HOTFIX

02.03 CONTEXTO MÍNIMO
El nombre debe indicar qué se hizo.

Evitar:

descripciones vagas
frases largas
lenguaje ambiguo

02.04 COMPATIBILIDAD IA
El formato debe ser:

fácilmente parseable
sin caracteres ambiguos
sin espacios innecesarios
03.00 FORMATO CANÓNICO GENERAL

Formato base:

yyyymmdd.HHmm_TYPE_SCOPE_description

Donde:

TYPE: tipo de cambio
SCOPE: área o componente
description: acción específica
04.00 COMMITS

04.01 FORMATO

yyyymmdd.HHmm_TYPE_SCOPE_description

04.02 EJEMPLOS

20260416.1135_INIT_STAGE6_baseline
20260416.1142_FIX_git_remote_alignment
20260416.1201_FEAT_excel_sensitivity_engine

04.03 REGLAS

Un commit = una intención clara
No mezclar múltiples cambios en un mismo commit
Evitar mensajes genéricos

04.04 ANTI-PATRONES

Incorrecto:

fix stuff
update
cambios varios
final ahora sí

05.00 BRANCHES

05.01 FORMATO

type/yyyymmdd.HHmm_description

05.02 TIPOS PERMITIDOS

feature
fix
hotfix
refactor
experiment

05.03 EJEMPLOS

feature/20260416.1205_sensitivity-matrix
fix/20260416.1210_git-push-error
hotfix/20260416.1215_tag-conflict

05.04 REGLAS

Usar kebab-case en description
No usar espacios
No usar nombres personales

05.05 ANTI-PATRONES

Incorrecto:

branch1
nuevo
pablo-prueba

06.00 TAGS

06.01 PROPÓSITO

Representan checkpoints inmutables del sistema.

06.02 FORMATOS

Nivel 1 — técnico (timestamp):

yyyymmdd.HHmm_CONTEXT

Ejemplo:
20260416.1135_STAGE6_BASELINE

Nivel 2 — semántico (release):

vMAJOR.MINOR.PATCH

Ejemplo:
v0.6.0

06.03 REGLAS

Un tag no se modifica una vez creado
Un tag representa un estado verificable
Un tag debe poder vincularse con BATON

06.04 ANTI-PATRONES

Incorrecto:

final
ultimo
test
stage6_final_real

07.00 RELACIÓN CON HIA

07.01 BATON

BATON referencia TAG
TAG representa estado técnico exacto

07.02 RADAR

RADAR puede indexar commits por timestamp
Permite trazabilidad automática

07.03 MULTI-IA

Permite reconstrucción de contexto
Reduce pérdida de continuidad
Facilita debugging entre sesiones
08.00 VARIACIONES PERMITIDAS

08.01 FORMATO COMPACTO

yyyymmdd.HHmm_TYPE_scope

Ejemplo:
20260416.1135_FIX_git

08.02 FORMATO EXTENDIDO

yyyymmdd.HHmm_TYPE_SCOPE_description_detallada

Ejemplo:
20260416.1135_FIX_git_remote_alignment_error

09.00 CRÍTICA A FORMATO SIMPLE

Formato propuesto original:

yyyymmdd.HHmm_Nombre

Problemas detectados:

no indica tipo
no permite clasificación
no escala en equipos
no permite automatización eficiente

Conclusión:

Debe evolucionar a:

yyyymmdd.HHmm_TYPE_SCOPE_description

10.00 RECOMENDACIÓN FINAL

Estándar obligatorio:

COMMITS:
yyyymmdd.HHmm_TYPE_SCOPE_description

BRANCHES:
type/yyyymmdd.HHmm_description

TAGS:
yyyymmdd.HHmm_CONTEXT
vMAJOR.MINOR.PATCH

11.00 RESULTADO ESPERADO

Aplicando este skill se obtiene:

repositorios auditables
trazabilidad temporal completa
compatibilidad con automatización IA
reducción de errores operativos
gobernanza técnica consistente
