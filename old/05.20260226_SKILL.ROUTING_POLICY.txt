==========
01.00_TASK_TAXONOMY
==========

Tipos mínimos:
- ARCH_DESIGN (arquitectura/filosofía/decisiones)
- CODE_GEN (crear código)
- REFACTOR (cambios multi-file)
- DEBUG (errores, logs, troubleshooting)
- DOC_CANON (documentos con FILE_CONTENT)
- DATA_XFORM (transformaciones simples)
- TOOLING (scripts, pipelines, automation)

==========
02.00_ROUTING_MVP_(SIMPLE)
==========

Modelo principal (cloud, alta calidad):
- M_MAIN = CLAUDE (o GPT-4o si decides)

Modelo secundario (manual):
- M_SECOND = GPT-4o (o Claude)

Regla:
- Todo va a M_MAIN.
- Solo usar M_SECOND cuando:
  - M_MAIN falla
  - o la tarea es CODE_GEN con requerimientos específicos

==========
03.00_ROUTING_ESCALABLE_(OPCIONAL)
==========

- ARCH_DESIGN -> M_MAIN
- REFACTOR -> M_MAIN
- DEBUG -> M_MAIN
- CODE_GEN -> M_SECOND si el costo lo permite
- DOC_CANON -> M_MAIN con contexto reducido
- DATA_XFORM -> modelo barato (si decides agregarlo)

==========
04.00_FALLBACK_(DEFINICION_Y_LIMITES)
==========

Fallback = cambio de modelo por error/latencia/calidad.

En MVP:
- fallback NO automático (para estabilidad).
- se registra recomendación: “usar M_SECOND” y el humano ejecuta.

En fase avanzada:
- fallback automático solo si:
  - logging + budgets + trazabilidad ya están maduros
