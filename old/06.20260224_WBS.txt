00.20_CONTEXTO_DEL_PROBLEMA_Y_OBJETIVOS
==========

----------
00.20.01_Problemas_reales_que_motivan_este_SKILL
----------

a) Rutas demasiado largas en Windows (≈260 caracteres) que rompen copias, zips,
   sincronización (OneDrive, etc.) y tooling.

b) Nombres extremadamente verbosos y heterogéneos que dificultan la lectura y
   el parseo automático.

c) Falta de compatibilidad entre nombres de FS (File System) y claves para BD
   (Base de Datos).

d) Refactors costosos porque versión, WBS, título y demás se mezclan en el nombre
   del archivo.

e) Inconsistencia entre proyectos y sistemas: cada uno inventa su propio esquema
   de nombres, obligando a RADAR y a la IA a “adivinar” patrones en vez de seguir
   reglas.

----------
00.20.02_Objetivo_principal
----------

Definir un estándar único y simple para:

a) WBS interno (estructura dentro de los archivos).
b) IDs cortos de archivos y carpetas (FS-friendly).
c) IDs para BD (DB-friendly).

----------
00.20.03_Objetivo_secundario
----------

Permitir que los nombres “humanos” puedan ser largos y explicativos sin romper:

a) Límites de Windows.
b) Esquemas de BD.
c) Scripts y tooling que dependan de los IDs.

==========
01.00_DEFINICIONES_BASICAS
==========

----------
01.00.01_WBS_INTERNO
----------

Estructura jerárquica dentro de un documento (por ejemplo: 01.00, 01.10, 01.20, etc.).

Función:

- Ordenar ideas y secciones de forma estable.
- Permitir referencias internas tipo “ver 03.20 de este documento”.

No tiene obligación de coincidir con la estructura de carpetas.

----------
01.00.02_ID_FS_(ID_de_File_System)
----------

Identificador corto y estable que se usa en nombres de archivos y carpetas.

Reglas mínimas:

- ASCII básico, sin espacios, sin tildes, sin caracteres raros.
- Amigable para Windows (ruta total razonable).
- Se compone normalmente de:

  - Prefijo de frame o proyecto (FRM).
  - Tipo lógico (TIPO).
  - Secuencia numérica (NNNN).
  - Opcional: sufijo humano corto en ASCII.

Ejemplo:

- PF_HUM_0005
- EVA_SKL_0101
- PF_TL_0200_REGENALL

----------
01.00.03_ID_DB_(ID_de_Base_de_Datos)
----------

Clave para BD:

- Lo más corta posible.
- Sin dependencias directas de rutas físicas.
- Idealmente unívoca para cada elemento lógico relevante
  (archivo, script, tabla lógica, sección, etc.).

Puede reutilizar el mismo patrón que ID_FS o derivarlo, pero siempre con
mapping claro ID_DB ↔ ID_FS.

----------
01.00.04_FRAME
----------

Conjunto coherente de reglas y artefactos:

- PFUNDATIONS, EVA.EOL, WINGS, KLAB, proyectos inmobiliarios, etc.

----------
01.00.05_SKILL
----------

Estándar transversal basado en errores reales que define “cómo se trabaja”.

Los frames deben respetar los SKILLS, salvo excepciones justificadas.

==========
02.00_PROBLEMAS_HISTORICOS_QUE_ESTE_SKILL_ATACA
==========

----------
02.00.01_Problema_1_Rutas_largas_en_Windows
----------

Árboles de carpetas profundos + nombres largos + OneDrive
= riesgo de superar el límite de ruta.

Síntomas:

- Errores al copiar/mover/eliminar.
- Imposibilidad de zippear.
- Fallos silenciosos de scripts.

----------
02.00.02_Problema_2_Nombres_bonitos_pero_inescalables
----------

- Espacios, tildes, ñ, caracteres especiales.
- Mezcla de WBS, título, versión y comentarios en el mismo string.

Resultado:

- Keys poco confiables en BD.
- Dificultad para generar índices/queries limpios.
- Dificultad para que la IA entienda patrones.

----------
02.00.03_Problema_3_Refactors_caros
----------

Cuando número de versión, WBS o nombre del documento cambian, el nombre
del archivo también cambia → cascada de roturas en scripts y referencias.

----------
02.00.04_Problema_4_Inconsistencia_entre_proyectos_y_sistemas
----------

Cada proyecto define su propia lógica de nombres.

- RADAR y tooling tienen que “adivinar” patrones.
- Reutilizar herramientas entre frames se vuelve más costoso.

==========
03.00_PRINCIPIOS_DE_DISENO_PARA_WBS_E_IDS
==========

----------
03.00.01_Principio_1_Separar_capas
----------

- WBS interno → lectura / organización conceptual.
- ID_FS       → integración con FS (nombres de archivo/carpeta).
- ID_DB       → integración con BD / catálogos.

No mezclar todo en el mismo string.

----------
03.00.02_Principio_2_ID_corto_semantica_larga
----------

- El ID debe ser mecánico, corto y estable.
- La semántica (texto explicativo) va en:

  - Títulos de sección.
  - Cabecera de archivos (metadatos).
  - Campos descriptivos de BD.

----------
03.00.03_Principio_3_ASCII_only_en_IDs
----------

En IDs de FS y BD:

- Letras A–Z (mayúsculas recomendadas).
- Dígitos 0–9.
- Separadores: _ y, cuando sea necesario, ..

No se permiten:

- Tildes, ñ, espacios, comillas, paréntesis, símbolos raros.

----------
03.00.04_Principio_4_Estabilidad_de_IDs
----------

Una vez asignado un ID_FS o ID_DB relevante:

- No se cambia salvo rediseño estructural mayor.
- Las versiones van en metadatos, no en el ID.

----------
03.00.05_Principio_5_Longitud_controlada
----------

Objetivo:

- ID_FS ≤ 16 caracteres (sin sufijo) recomendado.
- ID_FS + sufijo ≤ 40 caracteres.
- Nombre de archivo completo (ID_FS + sufijo + extensión) ≤ 80 caracteres.
- Ruta total objetivo: < 200 caracteres (buffer bajo límite Windows).

==========
04.00_ESTANDAR_DE_WBS_INTERNO_(DENTRO_DE_LOS_ARCHIVOS)
==========

----------
04.00.01_Formato_basico
----------

Formato sugerido:

NN.MM_TITULO_EN_MAYUSCULAS_SIN_TILDES

Ejemplo:

01.30_RELACION_CON_LOS_SKILLS_Y_ESTANDARES_DEL_SISTEMA

Características:

- NN = bloque principal (00, 01, 02, …).
- MM = subbloque (00, 10, 20, 30, etc.).
- Título en mayúsculas, sin tildes ni caracteres especiales.
- Guion bajo _ como separador entre código y título.

----------
04.00.02_Uso_recomendado
----------

- Ordenar el contenido y facilitar el referenceo.
- Usarlo en comentarios, discusiones, TROUBLE_LOG, documentación.

No se usa directamente como nombre de archivo ni como clave de BD.

----------
04.00.03_Reglas_de_estabilidad
----------

- Evitar renumerar WBS internos salvo rediseño profundo.
- Si se agrega un bloque nuevo intermedio, se puede usar:

  - 01.15, 01.25, etc., para evitar renumerar todo.

Mejor un WBS interno “feo pero estable” que perfecto y cambiante.

==========
05.00_ESTANDAR_DE_IDS_CORTOS_PARA_ARCHIVOS_Y_CARPETAS_(FS)
==========

++++++++++
05.00.01_Objetivo_de_ID_FS
++++++++++

Ser una clave corta, estable y legible que:

- Identifique unívocamente el recurso dentro de un frame.
- Sea parseable por scripts y tooling.
- Sea amigable con FS (rutas, zips, sincronización).

++++++++++
05.00.02_Patron_general_FRM_TIPO_NNNN
++++++++++

Patrón genérico:

FRM_TIPO_NNNN[_SUFIJO]

Donde:

- FRM  = código corto de frame/proyecto.
- TIPO = categoría lógica (HUM, SKL, TL, RAD, BTN, CFG, SCRP, KIT, etc.).
- NNNN = número secuencial de 4 dígitos (0001–9999).
- SUFIJO (opcional) = descriptor corto ASCII sin tildes.

++++++++++
05.00.03_FRM_oficiales_(ejemplos)
++++++++++

Ejemplos (incompletos, ampliables):

- PF    = PFUNDATIONS
- EVA   = EVA.EOL
- WNG   = WINGS
- KLAB  = Laboratorio Kumquat
- PFOP  = PFUNDATIONS.OPERATIONS (subframe)
- PROJX = Proyecto puntual X

El catálogo oficial de FRM se documenta en un HUMAN de arquitectura/taxonomía.

++++++++++
05.00.04_TIPOs_oficiales_(ejemplos)
++++++++++

Ejemplos de TIPO:

- HUM  = HUMAN.* (documentos explicativos para humanos)
- SKL  = SKILL / STANDARD
- TL   = TROUBLE / LOG / TOOL (según definición)
- RAD  = RADAR outputs
- BTN  = BATON.STATE y variantes
- CFG  = configuraciones
- SCRP = scripts (ps1, py, etc.)
- KIT  = kits (zip, bundles de entrega, etc.)

Este skill no congela la lista; solo establece la forma.

++++++++++
05.00.05_Ejemplos_de_ID_FS
++++++++++

a) PF_HUM_0005_OPERATIONS.txt

- Frame PFUNDATIONS (PF).
- Tipo HUMAN (HUM).
- Documento 0005 (OPERATIONS).

b) PF_TL_0101_REGENALL.ps1

- Script 0101 de tipo TOOL/LOG.

c) PF_KIT_0001_BOOTSTRAP_NEW.zip

- Kit 0001 de tipo KITS para proyectos nuevos.

++++++++++
05.00.06_Reglas_de_caracteres_y_longitud
++++++++++

Permitido:

- Letras A–Z, dígitos 0–9, _ y .

No permitido:

- Espacios, tildes, ñ, comillas, paréntesis, %, #, ?, *, /, \, etc.

Longitud:

- ID_FS (sin sufijo) ≤ 16 caracteres recomendado.
- Nombre de archivo completo ≤ 80 caracteres.

++++++++++
05.00.07_IDs_para_carpetas
++++++++++

Mismo patrón, pero sin extensión.

Ejemplos:

- PF_ROOT        → raíz del frame PFUNDATIONS.
- PF_HUM         → carpeta de HUMAN de PF.
- PF_TL          → carpeta de TOOLS de PF.
- PF_PROJX_0001_ProyectoIndustrialX → carpeta de un proyecto específico.

==========
06.00_ESTANDAR_DE_IDS_PARA_BD_(DB-FRIENDLY)
==========

++++++++++
06.00.01_Objetivo_de_ID_DB
++++++++++

Ser una clave estable y portátil que no dependa de:

- Ruta en disco.
- Nombre de archivo.
- Ubicación relativa.

++++++++++
06.00.02_Formato_general_para_documento/seccion
++++++++++

Formato sugerido:

FRM.DOC.SECC

Donde:

- FRM  = ID del frame (ej.: PFOP para PFUNDATIONS.OPERATIONS).
- DOC  = ID del documento dentro del frame,
         alineado con NNNN de ID_FS (ej.: 0005).
- SECC = ID de sección WBS (ej.: 0130 para 01.30; 0000 para “documento completo”).

Ejemplos:

- PFOP.0005.0000 → Documento completo HUMAN.OPERATIONS.
- PFOP.0005.0130 → Sección 01.30 de HUMAN.OPERATIONS.

++++++++++
06.00.03_Campos_adicionales_sugeridos_en_BD
++++++++++

Tabla de metadatos (ej. ARCHIVOS / OBJETOS):

- id_db        : clave primaria.
- id_fs        : nombre corto usado en FS.
- id_unico     : identificador lógico (ej. SKILL.WBS_ID.01.01).
- title_humano : título descriptivo.
- wbs_interno  : string tipo “01.30”.
- fs_path_rel  : ruta relativa dentro del frame.
- frame        : PF, EVA, WNG, etc.
- tipo_elemento: HUMAN, SKILL, TOOL, KIT, RADAR, ZIP, etc.

++++++++++
06.00.04_Reglas_de_caracteres_para_ID_DB
++++++++++

- Letras mayúsculas A–Z.
- Dígitos 0–9.
- Separador ..

Sin espacios, sin tildes, sin símbolos adicionales.

++++++++++
06.00.05_Relacion_ID_DB_↔_ID_FS_↔_WBS
++++++++++

- ID_FS referencia el archivo/carpeta físico.
- ID_DB referencia el “elemento lógico” (documento o sección).
- WBS ordena el contenido dentro del archivo.
- El mapping se mantiene en BD y/o en cabeceras de documentos.

==========
07.00_RELACION_IDS_CON_SKILLS_Y_HUMAN_Y_BATON
==========

----------
07.00.01_ID_UNICO_en_SKILLS_y_HUMAN
----------

Cada SKILL/HUMAN lleva un ID_UNICO lógico, por ejemplo:

- SKILL.FILE_CONTENT.01.01
- SKILL.WBS_ID.01.01
- HUMAN.PFUNDATIONS.README.01.00

Este ID_UNICO sirve como:

- Referencia documental estable.
- Identificador conceptual que no depende de dónde se guarde el archivo.

----------
07.00.02_ID_FS_en_SKILLS_y_HUMAN
----------

Los SKILLS y HUMAN usan además un ID_FS:

- 06.STANDAR.WBS_ID.txt        → ID_FS vinculado a SKILL.WBS_ID.01.01
- 07.STANDAR.FILE_CONTENT.txt  → ID_FS vinculado a SKILL.FILE_CONTENT.01.01
- HUMAN.PFUNDATIONS.README.txt → ID_FS alineado con HUMAN.PFUNDATIONS.README.01.00

Relación recomendada:

- ID_UNICO = clave lógica.
- ID_FS    = nombre corto del archivo.
- ID_DB    = clave en BD (puede coincidir con ID_FS o ser distinta, pero trazable).

----------
07.00.03_Referencias_entre_archivos
----------

Cuando un archivo hace referencia a otro, se recomienda usar:

- Nombre + ID_UNICO, por ejemplo:

  “Ver HUMAN.PFUNDATIONS.README (ID_UNICO: HUMAN.PFUNDATIONS.README.01.00).”

Esto evita romper referencias si el ID_FS cambia por algún motivo controlado.

----------
07.00.04_Relacion_con_BATON_y_RADAR
----------

- BATON.STATE usa ID_UNICO/ID_FS para listar HUMAN y SKILLS relevantes.
- RADAR_INDEX/CORE/FULL usa ID_FS para mapear archivos en el FS.
- BD usa ID_DB para mapear elementos lógicos que pueden vivir en uno o varios archivos.

Consistencia entre ID_UNICO, ID_FS e ID_DB es clave para trazabilidad.

09.00_ANTIPATRONES_(QUE_NO_HACER)
==========

----------
09.00.01_Antipatron_1_Meter_todo_en_el_nombre_de_archivo
----------

Ejemplo malo:

00.05_HUMAN.README.OPERATIONS.PFUNDATIONS_v0.3_definitivo_ahora_si.txt

Problemas:

- Largo, frágil, difícil de parsear, ridículo en BD.
- Imposible de gestionar en scripts sin hacks.

----------
09.00.02_Antipatron_2_Usar_IDs_distintos_en_FS_y_en_BD_sin_mapping
----------

FS:

- DocumentoOperacionesPF.txt

BD:

- PF_OPER_001

Sin tabla de mapping o metadatos claros, se rompe trazabilidad.

----------
09.00.03_Antipatron_3_Cambiar_IDs_al_cambiar_titulos
----------

No se debe renombrar:

- PF_HUM_0005_OPERATIONS.txt → PF_HUM_0006_OPERATIONS_v2.txt

solo porque cambiaste el contenido.

El ID debe ser estable; la versión va en metadatos, no en ID_FS.

----------
09.00.04_Antipatron_4_Mezclar_idiomas_y_acentos_en_IDs
----------

Ejemplo no aceptable:

- PF_HÚMAN_0005_OPERACIÓN.txt

Los IDs deben ser neutros, ASCII; el idioma lo lleva el contenido del documento.

10.00_IMPLEMENTACION_PROGRESIVA_(COMO_MIGRAR)
==========

----------
10.00.01_Regla_general
----------

No vale la pena renombrar todo lo antiguo de golpe.

Se migra cuando:

- Tocas un frame.
- Creas nuevos artefactos.
- Refactorizas tooling o BD.
- Un INCIDENT obliga a ordenar IDs.

----------
10.00.02_Estrategia_de_adopcion
----------

a) Nuevos documentos → siempre con ID_FS estándar.

b) Nuevos scripts → siempre con ID_FS estándar + WBS interno documentado.

c) Migración de documentos clave existentes cuando:

- Se actualicen de forma importante.
- Se incorporen a BD o a un nuevo RADAR.
- Se conviertan en referencias frecuentes para IA/humanos.

----------
10.00.03_Al_migrar_un_archivo_existente_a_ID_FS_nuevo
----------

a) Registrar el cambio en el historial del frame (HUMAN.HISTORY o similar).
b) Actualizar cualquier script que apunte a la ruta antigua.
c) Añadir una entrada en BD (tabla de mapping antiguo→nuevo si aplica).

11.00_CHECKLIST_RAPIDO
==========

----------
11.00.01_Para_un_archivo_o_carpeta
----------

- ¿Tiene ID_FS compatible con el patrón FRM_TIPO_NNNN[_SUFIJO]?
- ¿El nombre completo del archivo está razonablemente corto (<80 caracteres)?
- ¿El ID_FS es estable, o se renombra cada vez que cambias el título?
- ¿Se evita el uso de tildes, ñ y símbolos raros?

Si alguna respuesta es “no”, falta aplicar este skill.

----------
11.00.02_Para_un_elemento_en_BD
----------

- ¿Tiene un ID_DB que no dependa de la ruta física?
- ¿Existe mapping claro ID_DB ↔ ID_FS ↔ WBS?
- ¿Existe tabla de metadatos con título humano, path relativo y tipo_elemento?

Si alguna respuesta es “no”, falta aplicar este skill en la parte de BD.
