00.20.01_OBJETIVO_PRINCIPAL
- Diseñar un sistema estable para “ejecución frecuente” donde una IA:
  - entienda el proyecto,
  - proponga cambios,
  - cree/edite/elimine archivos SOLO dentro del proyecto,
  - haga rollback,
  - y controle costos automáticamente.

00.20.02_RESTRICCIONES_HUMANAS_Y_OPERATIVAS
- Prioridad: SIMPLICIDAD + ESTABILIDAD (no Frankenstein).
- Preferencia: CLOUD (modelos locales solo opcional / futuro).
- No usar web para trabajar: todo desde IDE + orquestador local.
- IA NO toca el OS “a voluntad”: solo FS dentro de sandbox.
- Pocas carpetas; muchos archivos; tamaño de archivos puede ser alto,
  con plan de segmentación (RADAR segs) cuando aplique.

00.20.03_SUPUESTOS
- Trabajo típico: proyectos propios (tamaño mediano).
- Windows 11.
- Git disponible (obligatorio para rollback real).

01.00_PRINCIPIOS_NO_NEGOCIABLES
==========

01.10_1_INTERFAZ_UNICA
- Un solo “front” para el trabajo diario (IDE principal).
- El chat web puede existir para diseño conceptual, pero NO es canal operativo.

01.20_1_ORQUESTADOR_LOCAL
- Un único proceso local “AI_ORCHD” que:
  - aplica sandbox,
  - aplica routing,
  - controla costos,
  - ejecuta agentes,
  - registra logs.

01.30_EJECUCION_ENCAPSULADA
- Agentes con capacidad de escribir/borrar SOLO dentro del PROJECT_ROOT.
- Cualquier intento fuera del root => FAIL + log (TROUBLE).

- Cada ejecución que modifique FS crea checkpoint:
  - Git commit (preferido) o snapshot interno (fallback).
- Sin rollback, no hay “ejecución frecuente segura”.

01.50_CONTRATO_DE_OK_VS_FAIL
- Todo runner/script/agente debe terminar con:
  - OK solo si outputs y validaciones mínimas se cumplieron.
  - FAIL si hubo error o violación de boundary.
- Ver skill PROBLEM_TROUBLE_INCIDENTS.

01.60_RADAR_ES_MAPA
- RADAR describe el FS, no decide.
- AI_ORCH consume RADAR para contexto controlado (INDEX default).

01.70_BATON_PARA_CONTINUIDAD
- BATON.STATE por proyecto: snapshot operativo para IA en blanco.
- BATON deriva de HUMAN + SKILLS + RADAR (no al revés).

03.00_CONTRATOS_DUROS_(INTERFACES)
==========

03.10_CONTRATO_IDE___ORCH
Entrada:
- TASK_REQUEST (JSON conceptual, no código):
  - objetivo
  - scope (módulos/paths)
  - constraints (no tocar X)
  - budget_hint (opcional)
Salida:
- TASK_PLAN (plan + estimate costo + riesgos)
- TASK_PATCH (diff/patch) o FILE_BUNDLE (archivos completos)
- TASK_RESULT (OK/FAIL + evidencia)

03.20_CONTRATO_ORCH___MODEL
- El orquestador manda:
  - contexto limitado (RADAR_INDEX por defecto)
  - snippets necesarios (selectivo)
- Recibe:
  - plan o patch
  - decisión de “needs more context”
Regla:
- “pedir más contexto” es válido; “asumir” no.

03.30_CONTRATO_ORCH___EXECUTOR
- Executor recibe instrucciones concretas:
  - crear archivo X con contenido Y
  - aplicar patch a archivo Z
- Executor valida:
  - path dentro sandbox
  - size/encoding
  - policy de delete (soft-delete)
- Executor reporta:
  - cambios (lista)
  - evidencias (paths + hashes)
  - OK/FAIL

03.40_CONTRATO_EXECUTOR___ROLLBACK
- Siempre existe rollback plan:
  - Git reset/checkout (preferido)
  - Snapshot restore (fallback)
- Rollback debe dejar evidencia.

==========

02.00_PERMISOS_Y_LIMITES_(FS)
==========

02.00_CONTRATO_DURO_DE_BOUNDARY
==========

02.10_REGLA_CENTRAL
- Toda operación FS debe pasar por una función:
  ResolveAndVerifyPath(path)
- Si ResolveAndVerifyPath detecta que el path resuelto NO está bajo PROJECT_ROOT:

00.20_OBJETIVO_Y_CRITERIOS

Objetivo:

- Mantener operación estable con ejecución frecuente.
- Proteger presupuesto cloud (evitar COST_INCIDENT).

Criterios:
- MVP: 1 modelo principal (M_MAIN) + 1 modelo secundario (M_SECOND) manual.
- No comparación simultánea.
- No fallback automático en MVP.
- Contexto por defecto: RADAR_INDEX (no CORE/FULL).

==========
01.00_TAXONOMIA_DE_TAREAS_(TASK_TYPE)
==========

Task types mínimos:

- ARCH_DESIGN   : arquitectura, filosofía, decisiones de diseño
- CODE_GEN      : crear código nuevo
- REFACTOR      : cambios multi-file / reorganización
- DEBUG         : errores, troubleshooting, lectura de logs
- DOC_CANON     : creación/edición de docs “serios” (FILE_CONTENT)
- TOOLING       : scripts/pipelines/automatización
- DATA_XFORM    : transformaciones simples (texto/CSV/etc.)
- OPS_GOV       : gobernanza operativa (BATON, RADAR policy, incident response)

Regla:
Todo request al orquestador debe indicar TASK_TYPE.
Si no se indica: default = ARCH_DESIGN.

==========
02.00_ROUTING_MVP_(ESTABLE)
==========

Definiciones:
- M_MAIN   = modelo principal (cloud)
- M_SECOND = modelo secundario (cloud)

Recomendación base (por calidad global en arquitectura/refactor):
- M_MAIN   = Claude (modelo actual principal que elijas)
- M_SECOND = GPT-4o (o equivalente)

Regla MVP:
- Todo va a M_MAIN.
- M_SECOND se usa SOLO cuando:
  - M_MAIN falla (error técnico/timeout)
  - el humano decide escalar por calidad en CODE_GEN específico
  - hay restricción de costo que recomienda alternativa (humano aprueba)

Esto maximiza estabilidad.

==========
03.00_ROUTING_ESCALABLE_(OPCIONAL,_FASE_2)
==========

Habilitar solo cuando:
- logging de costos está maduro
- budgets están funcionando
- el equipo/humano ya opera sin caos

Routing sugerido (si se habilita):
- ARCH_DESIGN -> M_MAIN
- REFACTOR    -> M_MAIN
- DEBUG       -> M_MAIN
- DOC_CANON   -> M_MAIN (con contexto reducido)
- CODE_GEN    -> M_SECOND cuando sea necesario
- TOOLING     -> M_MAIN o M_SECOND según costo/calidad
- DATA_XFORM  -> modelo barato (solo si decides agregar un tercer modelo)

Nota:
Agregar modelo barato solo si tienes COST_CONTROL real; si no, será otra fuente de drift.

==========
04.00_REGLAS_DE_FALLBACK_(MVP = MANUAL)
==========

Fallback = usar otro modelo por falla técnica o degradación de calidad.

MVP:
- No hay fallback automático.
- El orquestador puede:
  - registrar recomendación “TRY_M_SECOND”
  - guardar el prompt/contexto listo para reintentar
- El humano decide ejecutar el reintento.

Fase avanzada (opcional):
- fallback automático solo si:
  - está habilitado explícitamente
  - no rompe budgets
  - queda log completo (modelo A falló, modelo B se usó)

==========
05.00_REGLAS_DE_COSTO_Y_CONTEXTO_(ANTI-RUNAWAY)
==========

Reglas duras:
- Default context = RADAR_INDEX o RADAR_LITE.
- CORE solo por módulo objetivo (subset).
- FULL solo por auditoría puntual.

Reglas operativas:
- Preflight obligatorio para REFACTOR grande:
  - estimar tokens y costo (rango)
  - si excede umbral: dividir plan o pedir confirmación humana

Regla anti-loop:
- Max 5 iteraciones por tarea sin intervención humana.
- Si 3 fallas: crear TROUBLE (MEDIA) y parar.

Si se detecta COST_INCIDENT:
- elevar severidad según monto/impacto
- aplicar playbook anti-runaway

==========
06.00_REGLAS_PARA_AGENTES_EJECUTORES_(PLAN VS EJECUCION)
==========

El modelo (LLM) NO ejecuta directamente.
El modelo produce:
- PLAN (acciones de archivos)
El orquestador valida PLAN contra:
- sandbox boundary
- allowlist de comandos
- costo estimado
El executor aplica PLAN dentro de PROJECT_ROOT y produce evidencia.

Regla:
Si el modelo no tiene texto exacto de un archivo/bloque:
- prohibido “inventar” el replace
- pedir RADAR_CORE o extracto exacto
(anti-drift / anti-alucinación)

00.20_CONTEXTO_DEL_PROBLEMA_Y_OBJETIVOS
==========

00.20.01_PROBLEMAS_REALES_QUE_MOTIVAN_ESTE_SKILL

a) Rutas demasiado largas en Windows (≈260 caracteres) que rompen copias, zips,
sincronización (OneDrive, etc.) y tooling.

b) Nombres extremadamente verbosos y heterogéneos que dificultan la lectura y
el parseo automático.

c) Falta de compatibilidad entre nombres de FS (File System) y claves para BD
(Base de Datos).

d) Refactors costosos porque versión, WBS, título y demás se mezclan en el nombre
del archivo.

e) Inconsistencia entre proyectos y sistemas: cada uno inventa su propio esquema
de nombres, obligando a RADAR y a la IA a “adivinar” patrones.

00.20.02_OBJETIVO_PRINCIPAL

Definir un estándar único y simple para:

a) WBS interno (estructura dentro de los archivos).
b) IDs cortos de archivos y carpetas (FS-friendly).
c) IDs para BD (DB-friendly).

00.20.03_OBJETIVO_SECUNDARIO

Permitir que los nombres “humanos” puedan ser largos y explicativos sin romper:

a) Límites de Windows.
b) Esquemas de BD.
c) Scripts y tooling que dependan de los IDs.

01.00_DEFINICIONES_BASICAS
==========

01.00.01_WBS_INTERNO

Estructura jerárquica dentro de un documento (por ejemplo: 01.00, 01.10, 01.20, etc.).

Función:

Ordenar ideas y secciones de forma estable.

Permitir referencias internas tipo “ver 03.20 de este documento”.

No tiene obligación de coincidir con estructura de carpetas.

01.00.02_ID_FS_(ID_DE_FILE_SYSTEM)

Identificador corto y estable que se usa en nombres de archivos y carpetas.

Reglas mínimas:

ASCII básico, sin espacios, sin tildes, sin caracteres raros.

Amigable para Windows (ruta total razonable).

Se compone normalmente de:

Prefijo de frame o proyecto (FRM).

Tipo lógico (TIPO).

Secuencia numérica (NNNN).

Opcional: sufijo humano corto ASCII.

Ejemplo:

PF_HUM_0005

EVA_SKL_0101

WNG_TL_0200_REGENALL

01.00.03_ID_DB_(ID_DE_BASE_DE_DATOS)

Clave para BD:

Lo más corta posible.

Sin dependencias directas de rutas físicas.

Idealmente unívoca para cada elemento lógico relevante.

Puede reutilizar el patrón ID_FS o derivarlo, pero siempre con mapping claro ID_DB ↔ ID_FS.

01.00.04_FRAME

Conjunto coherente de reglas y artefactos: PFUNDATIONS, EVA.EOL, WINGS, proyectos, etc.

01.00.05_SKILL

Estándar transversal basado en errores reales que define “cómo se trabaja”.
Los frames deben respetar los SKILLS, salvo excepciones justificadas.

01.00.06_CATALOGO_FRM_(NUEVO, RECOMENDADO)

FRM recomendado (ejemplos; cada ecosistema mantiene su catálogo):
- PF   = PFUNDATIONS
- EVA  = EVA.EOL
- WNG  = WINGS
- KLAB = Kumquat Lab
- PROJ = Proyecto puntual
- OPS  = Operaciones / runbooks
- SYS  = Sistema base transversal

Regla:
- Si agregas FRM nuevo, documentarlo en el HUMAN root del ecosistema o en un catálogo central.

02.00_PROBLEMAS_HISTORICOS_QUE_ESTE_SKILL_ATACA
==========

02.00.01_PROBLEMA_1_RUTAS_LARGAS_EN_WINDOWS

Árboles profundos + nombres largos + sincronización = riesgo de superar límite de ruta.

Síntomas:

Errores al copiar/mover/eliminar.

Imposibilidad de zippear.

Fallos de scripts.

02.00.02_PROBLEMA_2_NOMBRES_BONITOS_PERO_INESCALABLES

Espacios, tildes, ñ, caracteres especiales.

Mezcla de WBS, título, versión en el mismo string.

Resultado:

Keys poco confiables en BD.

Dificultad para queries limpias.

Dificultad de parseo automático.

02.00.03_PROBLEMA_3_REFACTORS_CAROS

Cambias versión/título y el nombre de archivo cambia → cascada de roturas.

02.00.04_PROBLEMA_4_INCONSISTENCIA_ENTRE_SISTEMAS

Cada proyecto inventa su lógica.
RADAR y tooling deben “adivinar” patrones.

==========
03.00_PRINCIPIOS_DE_DISENO_PARA_WBS_E_IDS
==========

03.00.01_PRINCIPIO_1_SEPARAR_CAPAS

WBS interno → organización conceptual dentro del archivo.

ID_FS → integración con filesystem.

ID_DB → integración con BD/catálogos.

No mezclar todo en el mismo string.

03.00.02_PRINCIPIO_2_ID_CORTO_SEMANTICA_LARGA

El ID debe ser mecánico, corto y estable.

La semántica va en:

títulos de sección,

cabecera/metadatos,

campos descriptivos de BD.

03.00.03_PRINCIPIO_3_ASCII_ONLY_EN_IDS

En IDs FS y BD:

Letras A–Z (mayúsculas recomendadas).

Dígitos 0–9.

Separadores: _ y .

No se permiten:

tildes, ñ, espacios, símbolos raros.

03.00.04_PRINCIPIO_4_ESTABILIDAD_DE_IDS

Una vez asignado un ID:

No se cambia salvo rediseño estructural mayor.

Versiones van en metadatos, no en el ID.

03.00.05_PRINCIPIO_5_LONGITUD_CONTROLADA

Recomendaciones:

ID_FS ≤ 16 caracteres (sin sufijo).

ID_FS + sufijo ≤ 40 caracteres.

Nombre archivo (ID_FS + sufijo + ext) ≤ 80 caracteres.

Ruta objetivo < 200 caracteres.

==========
04.00_ESTANDAR_DE_WBS_INTERNO_(DENTRO_DE_LOS_ARCHIVOS)
==========

04.00.01_FORMATO_BASICO

Formato sugerido:
NN.MM_TITULO_EN_MAYUSCULAS_SIN_TILDES

Ejemplo:
01.30_RELACION_CON_LOS_SKILLS_Y_ESTANDARES_DEL_SISTEMA

Características:

NN = bloque principal (00, 01, 02, …).

MM = subbloque (00, 10, 20, 30, etc.).

Título en mayúsculas, sin tildes ni caracteres especiales.

Guion bajo _ como separador.

04.00.02_USO_RECOMENDADO

Ordenar contenido y facilitar referencia (“ver 03.20”).

No se usa como nombre de archivo ni como clave de BD.

04.00.03_REGLAS_DE_ESTABILIDAD

Evitar renumerar WBS internos salvo rediseño profundo.

Si agregas un bloque intermedio, usar 01.15, 01.25, etc.

++++++++++

Permitido:
A–Z, 0–9, _ y .

No permitido:
espacios, tildes, ñ, símbolos raros.

++++++++++

Ejemplos:

++++++++++
05.00.08_EXCEPCIONES_CONTROLADAS_A_ID_FS
++++++++++

Regla:
El estándar ID_FS es recomendado para consistencia y tooling.

Se permiten excepciones cuando el HUMANO define un contrato operativo explícito.

Ejemplo (real):
Script con nombre fijo: RADAR.ps1

Condiciones para permitir la excepción:

Documentar en metadatos del archivo:
“EXCEPCION_ID_FS...: SI (justificada)”

Motivo: contrato operativo humano / compatibilidad con tooling.

Objetivo:
Evitar drift silencioso.

++++++++++
05.00.09_VALIDACION_OPERATIVA_(NUEVO)
++++++++++

Regla:
Todo tooling/agente que cree nombres debe validar ID_FS.

Regex recomendado (mínimo):
^[A-Z0-9]{2,6}_[A-Z0-9]{2,6}_[0-9]{4}(_[A-Z0-9._]{1,30})?$

Checks:
- ASCII only
- longitud nombre archivo <= 80
- ruta total < 200 recomendado
- evitar duplicados (si existe catálogo)

Si falla:
- no crear archivo
- registrar TROUBLE (severidad BAJA/MEDIA según impacto)

++++++++++
05.00.09_IDS_PARA_ARTEFACTOS_DE_ORQUESTACION_(AI_ORCH)
++++++++++

Para artefactos del orquestador (logs, snapshots, reports), se recomienda:

- Mantener IDs cortos y parseables en nombres.
- Evitar timestamps en ID_FS (preferir timestamp en el sufijo del archivo).

Ejemplos:

AIOR_LOG_0001_RUNS\LOG.RUN.<RUN_ID>.txt
AIOR_CST_0001\COST_YYYYMMDD.ACTIVE.txt
AIOR_SNP_0001\SNAP-YYYYMMDD-HHMM-SEQ\manifest.txt

Objetivo:

Parseo por scripts + compatibilidad DB sin rutas imposibles.

04.40_CONTRATO_DE_EXIT_CODE_Y_ESTADO_FINAL
----------

OBLIGATORIO (CONTRATO):

Un script NO debe declarar éxito (OK/COMPLETADO) si hubo error no controlado.

Si ocurre una excepción no controlada:

Registrar error (Write-Error o equivalente).

Terminar con exit code != 0 (por defecto exit 1).

Imprimir FAIL (no OK) como estado final.

Solo se imprime “OK/COMPLETADO” si:

No hubo excepción,

Se generaron los outputs esperados (DONE binario),

Se dejó evidencia (rutas/archivos generados).

Objetivo:

Evitar falsos positivos (ej.: “OK” con fallos parciales).

Mejorar auditabilidad y confianza operacional.

----------
04.50_CONVENCIONES_POWERSHELL_(VERBS_APROBADOS)_(NUEVO)
----------

Problema real:
PowerShell emite warnings si una función/cmdlet usa verbos no aprobados:
"The cmdlet 'X' uses an unapproved verb."

Regla:
Preferir verbos aprobados por PowerShell (ejemplos comunes):
Get, Set, New, Remove, Add, Clear, Copy, Export, Import, Convert, Update,
Start, Stop, Invoke, Test, Confirm, Enter, Exit, Find, Format, Join,
Measure, Move, Out, Push, Pop, Read, Write.

Recomendación dura (para tu caso):
- Evitar: Validate-*
- Preferir: Test-* (validación)
- Preferir: Confirm-* (confirmación)
- Preferir: Invoke-* (ejecución/acción)

Ejemplo concreto:
- En vez de Validate-CISMetadata
  - Test-CISMetadata
  - Confirm-CISMetadata
  - Invoke-CISMetadataCheck

Nota:
Este skill no reemplaza documentación oficial de PowerShell; es una regla operacional anti-warning.

00.20_CONTEXTO_DEL_PROBLEMA_Y_OBJETIVOS
==========

00.20.01_PROBLEMA_REAL_QUE_ESTE_SKILL_ATACA

Cuando un problema se maneja “en caliente” sin método:

- Se pierde contexto.
- Se aplican cambios múltiples (“shotgun debugging”).
- Se declaran cosas como “OK” aunque hubo error.
- Se vuelve imposible reproducir y aprender.

Resultado típico:
La misma clase de error se repite (porque no hubo aprendizaje consolidado).

00.20.02_OBJETIVO_PRINCIPAL

Dar un método simple, repetible y auditable para:
Detectar → Registrar → Diagnosticar → Corregir → Validar → Aprender.

00.20.03_OBJETIVO_SECUNDARIO

Estandarizar el “contrato mínimo” de evidencia y cierre:
Si alguien más (o una IA en blanco) lee el TROUBLE/INCIDENT después,
puede entender qué pasó, por qué, y cómo se evitó repetirlo.

==========
01.00_PROPOSITO_Y_ALCANCE
==========

01.10_POR_QUE_EXISTE_ESTE_SKILL

Evitar que cada problema se aborde “desde cero” sin método.
Evitar shotgun debugging (cambiar todo a la vez).
Asegurar que los problemas relevantes queden registrados y reaprovechados como conocimiento.

Diferenciar entre:
- Problema puntual.
- Problema recurrente.
- Incidente serio con impacto alto.

01.20_A_QUE_ESCENARIOS_APLICA

Escenarios típicos:

- Errores técnicos (scripts, Excel, BD, entornos, permisos).
- Descalces de datos (flujos financieros, reportes).
- Problemas de procesos operativos (administración, documentación).
- Problemas de diseño (criterio equivocado, supuestos mal definidos).
- Problemas de IA: costos inesperados, loops, ejecución errónea, sandbox violado.

Si hay algo que “no anda bien” y merece más de 5 minutos de atención, este skill aplica.

==========
02.00_DEFINICIONES_BASICAS
==========

02.10_PROBLEM

Situación detectada que no necesariamente ha impactado nada aún, pero podría.
Es un desvío entre lo que debería pasar y lo que está pasando, observado al menos una vez.

02.20_TROUBLE

Problema concreto que afecta la operación de forma acotada y se registra en TROUBLE_LOG.

Una entrada TROUBLE_LOG es la unidad mínima de memoria de problemas del ecosistema.

02.30_INCIDENT

Evento con impacto relevante en operación, dinero, reputación o riesgo.

02.40_COST_INCIDENT_(NUEVO)

Subtipo de INCIDENT cuando:
- Se dispara el costo cloud (runaway tokens).
- Se ejecutan loops de IA que consumen presupuesto.
- Se usa contexto excesivo (ej.: RADAR_FULL) sin necesidad.
- Un agente ejecutor intenta acciones fuera de política (aunque se bloqueen).

Severidad típica: MEDIA a CRITICA según monto y riesgo.

03.20_SEVERIDAD_(EJEMPLOS)
++++++++++

BAJA:
- Bug menor sin impacto en dinero/decisiones, workaround fácil.

MEDIA:
- “OK con error” (falso positivo) en scripts operativos (mínimo MEDIA).
- Loop leve de IA que sube costo pero se detecta rápido.

ALTA:
- Error en flujo financiero o automatización que bloquea trabajo relevante.
- Violación de sandbox (aunque haya sido DENY) con riesgo de configuración/política.

CRITICA:
- Error usado en decisiones de inversión de terceros.
- Exposición de secreto.
- Runaway cost que quema presupuesto significativo.

++++++++++
03.30_SEVERIDAD_A_ACCIONES_MINIMAS_(CONTRATO_OPERATIVO)
++++++++++

BAJA:
- Registrar TROUBLE_LOG mínimo.
- Workaround aceptable.

MEDIA:
- Parche mínimo (cambio único y localizado).
- Evidencia antes/después.
- Smoke test.
- Changelog en artefacto afectado (si aplica).

ALTA:
- Parche + validación extendida.
- Evaluar rollback plan.
- Evaluar regenerar RADAR.
- Actualizar al menos 1 SKILL/HUMAN si el aprendizaje es generalizable.

CRITICA:
- Freeze temporal del flujo afectado.
- Postmortem breve (causa raíz + prevención).
- Revisión humana obligatoria.
- Regenerar RADAR y evaluar actualizar BATON.STATE si aplica.

++++++++++
03.20.10_EJEMPLOS_AI_ORCH_(BOUNDARY_Y_SECRETOS)
++++++++++

Ejemplo A: Boundary violation (sandbox)

- TIPO: TROUBLE
- SEVERIDAD: MEDIA (mínimo)
- Evento: agente intentó escribir fuera de PROJECT_ROOT.
- Acción: FAIL inmediato + log + revisar SANDBOX_BOUNDARY.

Ejemplo B: Exposición de secreto

- TIPO: INCIDENT
- SEVERIDAD: ALTA (mínimo)
- Evento: API key/secret aparece en repo o logs.
- Acción: rotar key, limpiar evidencia, bloquear apply, postmortem breve.

04.10_DE_PROBLEM_A_TROUBLE
++++++++++

04.20_CUANDO_ES_INCIDENT
++++++++++

Un TROUBLE se eleva a INCIDENT cuando:
- tiene impacto alto o riesgo real (dinero, reputación, cumplimiento)

04.40_CONTRATO_DE_SALIDA_PARA_SCRIPTS
==========

++++++++++
04.40.01_REGLA_BINARIA_OK_VS_FAIL
++++++++++

Para scripts ejecutables (ps1, py, sh, sql runners, etc.):

Si ocurre una excepción NO controlada, o estado inválido:
- el script termina en FAIL
- imprime FAIL (no OK)
- retorna exit code != 0 (idealmente 1)

Solo se permite imprimir OK si:
- no hubo excepciones
- outputs esperados existen
- validación mínima pasa (existencia + tamaño > 0 + checks simples)

++++++++++
04.40.02_EVIDENCIA_MINIMA_OBLIGATORIA_EN_FAIL
++++++++++

Un FAIL debe registrar al menos:
- comando exacto ejecutado (copy/paste)
- path del script ejecutado
- versión del script (VERSION en metadatos)
- error completo (mensaje y stacktrace)
- outputs parciales creados (si existen) y rutas exactas

++++++++++
04.40.03_VALIDACION_MINIMA_EN_OK
++++++++++

Un OK debe verificar:
- outputs esperados existen
- tamaño > 0
- logs escritos
- DONE binario (si aplica)


++++++++++
04.50_POWERSHELL_VERBOS_APROBADOS_(EVITAR_WARNINGS)
++++++++++

Regla:

Evitar verbos no aprobados en funciones/cmdlets PowerShell
(ej: "Validate-*") para prevenir warnings tipo:

"The cmdlet 'Validate-CISMetadata' uses an unapproved verb."

Preferir verbos aprobados:

Test-*
Get-*
Set-*
New-*
Remove-*
Invoke-*
Export-*
Import-*
Measure-*

Ejemplo:

- En vez de Validate-CISMetadata -> Test-CISMetadata
- En vez de Validate-ProjectPaths -> Test-ProjectPaths

Objetivo:

Evitar warnings, mejorar compatibilidad con tooling y políticas corporativas.

06.30_PLAYBOOK_TECNICO_ANTI_SHOTGUN
==========

06.30.01_REGLA_1_UN_CAMBIO_POR_ITERACION
- una hipótesis única
- un cambio mínimo
- prueba antes/después
- evidencia

06.30.02_REGLA_2_REPRODUCIR_ANTES_DE_TOCAR
- reproducir con comando exacto
- capturar output/stacktrace
- registrar entorno (OS, path, permisos, versiones)

06.30.03_REGLA_3_VALIDAR_TIPOS_Y_SUPUESTOS
- confirmar tipo real (string vs list/dict)
- normalizar antes de operar
- no llamar métodos “asumidos” sin check

06.30.04_REGLA_4_PATCH_POR_BLOQUES
- entregar BLOQUE_ORIGINAL y BLOQUE_CORREGIDO
- parche local (no reescritura total) salvo necesidad

06.30.05_REGLA_5_NO_DECLARAR_OK_SI_HUBO_ERROR
- “OK con error” = severidad MEDIA mínimo

06.30.06_REGLA_6_SMOKE_TEST_MINIMO
- correr 1 caso feliz (happy path)
- correr 1 caso que antes fallaba
- validar outputs (existencia + tamaño > 0)

==========
06.35_PLAYBOOK_ANTI_RUNAWAY_(IA/COSTO)_(NUEVO)
==========

Objetivo:
Evitar loops de IA que consumen presupuesto o generan drift.

Reglas:
- Limitar iteraciones: max 5 reintentos por tarea sin intervención humana.
- Si 3 reintentos fallan: elevar a TROUBLE (MEDIA) y parar.
- Si el costo del día excede umbral: HARD_STOP para tareas no P0.
- Preferir RADAR_INDEX por defecto; CORE solo por módulo; FULL ocasional.

Evidencia mínima en COST_INCIDENT:
- estimación vs real
- tokens (input/output)
- task_type
- qué contexto se usó (INDEX/CORE/FULL)
- por qué se disparó (loop, prompts, bug)

00.20_CONTEXTO_DEL_PROBLEMA_Y_OBJETIVOS
==========

00.20.01_PROBLEMAS_REALES_QUE_MOTIVAN_RADAR

a) Proyectos crecen y se vuelven inmanejables sin inventario confiable.
b) Tooling e IA fallan cuando el repositorio no tiene mapa.
c) Auditorías, migraciones y refactors requieren trazabilidad.
d) Archivos binarios (xlsx/pdf/png) rompen intentos de “leer todo”.
e) Históricos sin disciplina generan caos; sin históricos no hay continuidad.
f) En IA cloud, subir FULL/CORE completo puede disparar costos (COST_INCIDENT).

00.20.02_OBJETIVO_PRINCIPAL

Definir un estándar único para producir vistas:

LITE : Inventario resumido + diffs básicos (operación diaria).
INDEX: Inventario total + metadata (para navegación y auditoría).
CORE : Contenido legible (texto/código) para análisis por IA/humano.
FULL : Consolidación (INDEX + CORE + extras de auditoría como tree-size).

00.20.03_OBJETIVO_SECUNDARIO

Asegurar que los outputs puedan:

Ser usados por humanos (lectura rápida, texto plano).

Ser consumidos por IA (chunks y determinismo).

Ser comparables entre ejecuciones (orden estable + formato consistente).

Reducir fricción/costo en cloud evitando “leer todo” sin necesidad.

01.00_DEFINICIONES_BASICAS
==========

01.10_RADAR

RADAR es un estándar de “descripción del FS” (file system) de un proyecto, generado por tooling.

Regla dura:

RADAR describe, no decide.

RADAR no reemplaza decisiones humanas ni lógica de negocio.

----------
01.15_RADAR_LITE_(NUEVO)
----------

RADAR_LITE es una salida liviana para operación diaria:

- Basada en metadata (similar a INDEX).
- Puede incluir diffs básicos vs ejecución previa (NEW/MODIFIED/DELETED).
- No incluye contenido (no es CORE).
- No interpreta semántica (describe, no decide).

01.20_RADAR_INDEX

Output que lista TODOS los archivos/carpetas relevantes y su metadata (sin contenido).

Incluye, típicamente:

ruta

tamaño

última modificación

tipo lógico

hash (si aplica)

01.30_RADAR_CORE

Output que incluye el contenido SOLO de archivos legibles (texto/código/config).

No incluye:

imágenes, audio, video

pdf (sin OCR)

binarios

xlsx (no se leen celdas)

01.40_RADAR_FULL

Output que consolida:

INDEX + CORE

y agrega extras de auditoría (ej.: tree-size por carpeta)

FULL no agrega semántica de negocio; sigue siendo descriptivo.

02.40_MODO_LOCAL_FS_VS_MODO_AI_UPLOAD
++++++++++

MODO_LOCAL_FS (máquina local / FS completo):

No hay límite de tamaño por archivo fuente para CORE (según versión base).

Se genera output ACTIVE completo.

Si ACTIVE > 8 MB:

se mantiene el ACTIVE intacto,

se generan copias segmentadas <= 8 MB para IA/humano.

MODO_AI_UPLOAD (subida a IA/chat):

Se respeta límite de tamaño por tanda (ej. 8–9MB).

Se suben segmentos (chunks) al chat para consumo por IA.

El activo completo puede no subirse (pero existe localmente).

Regla operativa recomendada:

- Default diario: RADAR_LITE o RADAR_INDEX.
- CORE: solo cuando vas a analizar/editar un módulo específico.
- FULL: ocasional (auditoría o “foto completa”), no por costumbre.

Si usas IA cloud:
- Evitar subir FULL completo al chat salvo necesidad real.
- Preferir subir segmentos CORE del módulo objetivo.

++++++++++
02.50_NOTA_SOBRE_CONSUMIDORES_(NO_PARTE_DEL_CONTRATO)
++++++++++

RADAR puede ser consumido por sistemas externos (ej: orquestadores IA)
para navegación y selección de contexto.

Importante:

- Esto NO cambia el contrato RADAR.
- RADAR sigue describiendo, no decidiendo.
- RADAR no depende de consumidores.

Recomendación para consumidores:

- Usar RADAR_INDEX como default.
- Escalar a CORE por módulo.
- FULL solo auditoría.

00.20_CONTEXTO_Y_OBJETIVOS
==========

BATON existe porque:
- Trabajar con IA (y con humanos) sin snapshot operativo produce pérdidas de continuidad.
- Un “HUMAN” explica el por qué; un “RADAR” describe el FS; un “SKILL” define método.
- Pero al levantar una IA en blanco, el primer problema real es: “¿Qué cargo y en qué orden?”

Objetivo principal:
- Permitir que una IA en blanco tenga continuidad operativa del proyecto en 1–2 minutos de lectura.

Objetivo secundario:
- Evitar drift: BATON no redefine reglas; solo referencia lo que ya es canon (HUMAN/SKILLS/RADAR).

==========
01.00_PROPOSITO_Y_ROL_DE_BATON
==========

----------
01.10_QUE_ES_BATON_STATE
----------

BATON.STATE es un archivo de texto (.txt) por proyecto que:

- Resume el estado operativo relevante para una IA en blanco.
- Indica qué HUMAN y SKILLS son canon en ese proyecto.
- Indica qué RADAR usar como mapa de archivos.
- Expone objetivos activos, contexto mínimo y focos de trabajo.

BATON.STATE es continuidad operativa para IA, nada más.
No crea reglas nuevas de negocio ni de código.

----------
01.20_POR_QUE_ES_POR_PROYECTO
----------

Se define un BATON.STATE por proyecto (no uno global) porque:

- Cada proyecto tiene objetivos, artefactos y riesgos propios.
- Los HUMAN y SKILLS “core” por proyecto no son siempre los mismos.
- La continuidad que te importa cuando levantas una IA es la del proyecto en cuestión.

Ejemplos de “proyecto”:

- PFUNDATIONS (marco personal/profesional).
- EVA.EOL (evaluación y aprendizaje).
- Proyecto inmobiliario específico.
- Sub-sistema técnico con vida propia.

----------
01.30_PRECEDENCIA_DE_FUENTES
----------

Orden de precedencia cuando hay conflicto:

1) HUMAN.README de nivel superior.
2) HUMAN.README específico del proyecto.
3) SKILLS/estándares (incluyendo este BATON).
4) BATON.STATE del proyecto.
5) Código/scripts/configuración.

Si BATON.STATE contradice HUMAN o SKILLS, BATON.STATE está mal
y debe regenerarse.

BATON.STATE nunca es “fuente de verdad funcional”; es derivado consolidado.

----------
01.40_RELACION_CON_ORQUESTACION_IA_(NUEVO)
----------

Si el proyecto usa agentes que crean/editar/eliminan archivos:
- BATON.STATE debe declarar explícitamente el “límite operativo” del agente.
- BATON.STATE NO define políticas nuevas, pero SÍ debe apuntar a los SKILLS que las definen.

En otras palabras:
- BATON no inventa sandbox/costos/routing.
- BATON referencia qué política aplica en este proyecto.

==========
02.00_CONTENIDO_MINIMO_DE_BATON_STATE
==========

++++++++++
02.10_BLOQUES_OBLIGATORIOS
++++++++++

Cada BATON.STATE debe incluir, como mínimo:

- Metadatos estándar del archivo (según FILE_CONTENT).
- Identificación clara del proyecto (nombre, ID, ruta raíz lógica).
- Lista de HUMAN relevantes (root e hijos) con sus rutas.
- Lista de SKILLS clave que aplican a ese proyecto.
- Referencias a RADAR (INDEX/CORE/FULL) vigentes.
- Objetivos/FOCOS activos del proyecto.
- Estado de avance/resumen ejecutivo muy breve.
- Checklist de “cómo iniciar la IA” para ese proyecto.

++++++++++
02.20_EJEMPLO_DE_SECCION_DE_CONTEXTO
++++++++++

Un bloque típico de BATON.STATE debería ser capaz de contestar, en pocas líneas:

- “Este proyecto trata de…”
- “Objetivo principal en este momento…”
- “Prioridades P0/P1/P2 activas…”
- “Riesgos o puntos sensibles que la IA debe tratar con cuidado…”

Si BATON.STATE no ayuda a una IA en blanco a responder esto en 1–2 minutos de lectura,
el archivo está incompleto.

++++++++++
02.30_REFERENCIAS_A_HUMAN_Y_SKILLS
++++++++++

BATON.STATE debe listar explícitamente:

- HUMAN root del proyecto.
- HUMAN hijos relevantes (si el HUMAN root fue partido).
- SKILLS que son críticos para operar aquí (ej.: FILE_CONTENT, PROBLEM_TROUBLE, RADAR, DRAG_AND_DROP, etc.).

No es necesario copiar el contenido de HUMAN/SKILLS.
Basta con referenciar nombre de archivo y ubicación lógica.

++++++++++
02.40_REFERENCIAS_A_RADAR
++++++++++

BATON.STATE debe indicar:

- Qué archivo RADAR_INDEX usar como mapa rápido.
- Qué archivo RADAR_CORE y/o RADAR_FULL está vigente.
- Dónde se ubican los históricos de RADAR (para auditoría si aplica).

BATON.STATE no duplica RADAR; solo dice “usa este RADAR como mapa”.

++++++++++
02.50_LIMITES_OPERATIVOS_DE_IA_(NUEVO, RECOMENDADO)
++++++++++

Si el proyecto usa IA ejecutora (que toca archivos):
BATON.STATE debería incluir un bloque “LIMITES_OPERATIVOS” con:

- PROJECT_ROOT (ruta raíz lógica del proyecto).
- Qué carpetas están habilitadas para escritura por el agente.
- Qué carpetas son read-only por política.
- Regla dura: “fuera de PROJECT_ROOT = prohibido”.

Este bloque NO reemplaza el skill de sandbox/política;
solo deja el puntero y el boundary operativo visible al inicio.

++++++++++
02.60_COSTOS_Y_USO_CLOUD_(NUEVO, RECOMENDADO)
++++++++++

Si el proyecto usa IA cloud con costo variable:
BATON.STATE debería incluir:

- Presupuesto mensual objetivo (si existe).
- Umbral de stop (si existe).
- Política de “default context”: usar RADAR_INDEX por defecto.

De nuevo:
- BATON no inventa esto.
- BATON referencia el skill/política aplicable o el contrato operativo acordado.

==========
03.00_NOMBRE_Y_FORMATO_DE_BATON_STATE
==========

----------
03.10_FORMATO_DE_ARCHIVO
----------

BATON.STATE es siempre un `.txt` que sigue el skill FILE_CONTENT:

- Bloque de metadatos al inicio.
- Introducción corta.
- Secciones ordenadas por títulos 1–4.
- Checklist de uso.
- CHANGELOG mínimo.

Nada impide que el contenido sea rico y largo, pero el formato base debe respetarse.

----------
03.20_NOMBRADO_SUGERIDO
----------

Sugerencia de nombre físico:

- `00.0_BATON_<Proyecto>.txt` dentro de la carpeta raíz del proyecto.
- ID_UNICO alineado con WBS_ID del proyecto.

Ejemplos:

- `00.0_BATON_PFUNDATIONS.txt`
- `00.0_BATON_EVA_EOL.txt`
- `00.0_BATON_PROY_WHISPERING_LAKES.txt`

Lo importante es que:

- Sea evidente que es BATON.STATE.
- Sea fácil de encontrar por humanos y por scripts.

----------
03.30_CONTENIDO_RICO_PERO_NO_CAOTICO_(NUEVO)
----------

BATON puede ser largo si aporta valor, pero debe seguir este control:

- P0: “cómo iniciar la IA” (máximo 1 página).
- P1: “objetivos/riesgos activos”.
- P2: “referencias” (HUMAN/SKILLS/RADAR).
- P3: “contexto adicional” (solo si evita errores reales).

Si BATON se vuelve un manual técnico: está mal. Eso va a SKILLS/TOOLS.

==========
04.00_USO_DE_BATON_POR_IA
==========

++++++++++
04.10_LOAD_ORDER_TIPICO
++++++++++

Cuando levantas una IA en blanco para trabajar en un proyecto:

1) Cargar BATON.STATE del proyecto.
2) Desde BATON.STATE, identificar:
   - HUMAN root + HUMAN hijos relevantes.
   - SKILLS clave.
   - RADAR_INDEX/CORE/FULL vigentes.
   - Objetivos activos.

3) Cargar HUMAN y SKILLS que BATON.STATE referencia.
4) Usar RADAR para navegar archivos del proyecto.

Si BATON.STATE no permite ejecutar este flujo, el archivo debe mejorarse.

++++++++++
04.20_QUE_PUEDE_PEDIR_LA_IA_A_BATON
++++++++++

La IA puede usar BATON.STATE para:

- Entender “qué hay en juego” en el proyecto.
- Saber qué archivos puede tocar y cuáles no (si el proyecto lo declara).
- Determinar qué HUMAN/SKILLS tienen prioridad.
- Ubicar rápidamente artefactos en el FS usando RADAR.

La IA no debe:

- Modificar BATON.STATE por iniciativa propia.
- Alterar HUMAN/SKILLS basándose solo en lo que dice BATON.STATE.

Cualquier cambio en BATON.STATE debe ser validado por humano.

++++++++++
04.30_COMO_USAR_BATON_CON_AGENTES_EJECUTORES_(NUEVO)
++++++++++

Si la IA ejecuta planes (crear/editar/eliminar):

- BATON.STATE debe indicar:
  - si hay política de ejecución (EXEC_POLICY) y sandbox
  - si hay contratos de rollback (git checkpoints)
  - si hay política de costos (budgets / hard-stop)

La IA debe:
- leer ese bloque antes de ejecutar cambios
- en caso de duda: pedir confirmación humana (no actuar a ciegas)

++++++++++
04.30_BATON_Y_AI_ORCH_(ORQUESTACION_Y_EJECUCION)
++++++++++

BATON.STATE debe incluir (cuando el proyecto use AI_ORCH):

- PROJECT_ROOT (ruta lógica del proyecto).
- Referencia a SANDBOX_BOUNDARY activo (nombre/ubicación).
- Referencia a AGENT_EXECUTION_POLICY activo.
- Referencia a COST_CONTROL y ROUTING_POLICY activos.
- Referencia a LOGGING_AUDIT y CHECKPOINT_ROLLBACK si existen.

Objetivo:

Que una IA en blanco sepa:
- qué puede tocar (y qué no),
- cómo se ejecuta con rollback,
- cómo se controla costo.

Regla:

BATON.STATE NO duplica políticas; solo referencia los skills canónicos.

05.00_CUANDO_REGENERAR_BATON_STATE

----------
05.10_EVENTOS_QUE_DISPARAN_REGENERACION
----------

BATON.STATE debería regenerarse cuando ocurran cambios como:

- Cambio relevante en objetivos o prioridades del proyecto.
- Creación o gran refactor de HUMAN root o SKILLS clave.
- Reestructuración importante de carpetas y archivos (RADAR cambia).
- Cierre de una fase importante y apertura de otra.

No hace falta regenerar BATON.STATE por cada typo o ajuste menor.

----------

Si un INCIDENT (según PROBLEM_TROUBLE_INCIDENTS) implica:

- Cambio profundo en forma de operar el proyecto.
- Cambio en qué archivos/skills son críticos.

Entonces:

- Se debe evaluar actualizar HUMAN/SKILLS primero.
- Luego se regenera BATON.STATE para reflejar la nueva realidad.

En TROUBLE_LOG puedes dejar referencia explícita al BATON.STATE regenerado.

05.30_CAMBIOS_DE_POLITICA_DE_EJECUCION_O_COSTO_(NUEVO)
----------

Si cambia cualquiera de estos puntos:
- boundary del sandbox
- reglas de “qué puede tocar el agente”
- budgets de costo cloud
- modelo principal (routing)

Entonces:
- se considera cambio operativo relevante
- regenerar BATON.STATE (porque afecta continuidad y riesgo)

02.00_HUMAN_ROOT_VS_HUMAN_HIJO
==========

++++++++++
02.10_HUMAN_ROOT
++++++++++

HUMAN root:

- Es el HUMAN principal de un sistema/proyecto.
- Define el alcance, objetivos, principios y lenguaje común.
- Es la referencia superior en caso de conflicto de interpretación.

Solo debería haber un HUMAN root por sistema/proyecto.

++++++++++
02.20_HUMAN_HIJO
++++++++++

HUMAN hijo:

- Es un HUMAN derivado, más específico.
- Puede concentrarse en un subdominio, módulo, fase o vista particular.
- No puede contradecir al HUMAN root.

Ejemplos:

- HUMAN.PFUNDATIONS.README (root).
- HUMAN.PFUNDATIONS.CAPAS (hijo focalizado en arquitectura).
- HUMAN.PROYECTO_X.FINANZAS (hijo de un HUMAN root del proyecto X).

++++++++++
02.30_CUANDO_PARTIR_UN_HUMAN
++++++++++

Un HUMAN root se parte en HUMAN hijos cuando:

- Es demasiado largo para ser operable.
- Mezcla dominios que podrían vivir separados.
- Hay lectores/usuarios claramente distintos (ej.: negocio vs técnico).

Regla práctica:

- Si un HUMAN supera el punto en que nadie quiere leerlo completo,
  se evalúa partir en HUMAN hijos, manteniendo un HUMAN root más estable.

03.00_JERARQUIA_Y_RESOLUCION_DE_CONFLICTOS
==========

----------
03.10_PRECEDENCIA_ENTRE_HUMAN
----------

Orden de precedencia cuando HUMAN chocan entre sí:

1) HUMAN root de nivel más alto (ej.: HUMAN.PFUNDATIONS.README).
2) HUMAN root del proyecto específico.
3) HUMAN hijos de ese proyecto/sistema.

Si dos HUMAN root de nivel similar chocan, se debe resolver explícitamente
en una siguiente versión (no se “deja pasar”).

----------
03.20_RELACION_HUMAN_CON_SKILLS_Y_BATON
----------

Precedencia global:

1) HUMAN (root e hijos, respetando jerarquía entre ellos).
2) SKILLS/estándares técnicos.
3) BATON.STATE (snapshot operativo por proyecto).
4) Código/scripts/configuración.

Si un SKILL contradice HUMAN, hay dos opciones:

- El SKILL está desactualizado → se actualiza.
- HUMAN está desactualizado → se actualiza, dejando claro el cambio.

BATON.STATE solo empaqueta el estado que HUMAN y SKILLS ya definieron.

----------
03.30_RELACION_CON_RADAR
----------

RADAR:
- Describe el filesystem de un proyecto en forma determinista.
- No decide arquitectura ni negocio.

HUMAN:
- Puede referenciar RADAR como “mapa”.
- No debe copiar outputs RADAR dentro del HUMAN (se referencian).

00.20_CONTEXTO_Y_OBJETIVOS
==========

Problema real:
- En chats con IA, archivos largos se truncan o se interpretan parcial.
- Se pierde contexto por “asumir” que el archivo estaba completo.
- Se mezclan versiones: se edita una copia vieja sin darse cuenta.

Objetivo principal:
- Asegurar que el flujo “subo archivo -> IA lo entiende -> IA devuelve cambios aplicables”
sea repetible, auditable y sin ambigüedad.

Objetivo secundario:
- Permitir trabajo en muchos archivos grandes sin volverse loco con carpetas profundas.

==========
01.00_DEFINICIONES
==========

DRAG_AND_DROP:
- Acción de adjuntar un archivo al chat/IA.

TANDA:
- Segmento de entrega cuando el contenido no cabe o se decide separar por operabilidad.

PAQUETE:
- Conjunto de archivos subidos para una tarea (ej.: HUMAN + SKILLS + RADAR segs).

TRUNCACION:
- El sistema corta el contenido (visible o invisible) y la IA ve solo una parte.

CONFIRMACION_DE_RECEPCION:
- Respuesta obligatoria de la IA listando EXACTAMENTE qué archivos/segmentos recibió.

==========
02.00_PRINCIPIOS_DE_DRAG_AND_DROP
==========

02.10_PRINCIPIO_1_NO_ASUMIR_COMPLETITUD
- Si hay duda de truncación, se trata como “NO completo” hasta verificar.

02.20_PRINCIPIO_2_TRAZABILIDAD
- Cada archivo tiene ID_UNICO + VERSION + FECHA.
- La IA debe citar qué versión está usando.

02.30_PRINCIPIO_3_TANDAS_EXPLICITAS
- Si se divide, se divide con contrato:
  - TANDA X/Y
  - nombre de archivo
  - qué hacer con la tanda (reemplazar / concatenar / agregar)

02.40_PRINCIPIO_4_CERO_DRIFT
- Si la IA no tiene el bloque exacto, no hace replace “a ojo”.
- Pide el bloque faltante o un extracto.

==========
03.00_PROTOCOLO_DE_CARGA_DE_ARCHIVOS_(UPLOAD)
==========

++++++++++
03.10_PASO_0_PREPARACION
++++++++++

Antes de subir:
- Verifica que el archivo tenga metadatos (FILE_CONTENT).
- Verifica VERSION y FECHA.
- Verifica que el nombre físico coincida con NOMBRE_SUGERIDO (si aplica).

++++++++++
03.20_PASO_1_SUBIR_ARCHIVOS
++++++++++

Subir al chat/IA:
- 1 a 6 archivos por turno idealmente (operabilidad).
- Si son más: agrupar por “paquetes” (ej. paquete_skills_core).

++++++++++
03.30_PASO_2_CONFIRMACION_OBLIGATORIA_DE_LA_IA
++++++++++

La IA debe responder inmediatamente con:

- LISTA_DE_ARCHIVOS_RECIBIDOS:
  - nombre físico exacto
  - ID_UNICO
  - VERSION
  - FECHA (si está en metadatos)
- NOTA_DE_INTEGRIDAD:
  - “No detecto truncación” o “Hay riesgo de truncación”
- BLOQUE_DE_DEPENDENCIAS:
  - si falta un archivo dependiente, lo declara

Si la IA no puede confirmar, se asume incompleto.

++++++++++
03.40_PASO_3_MAPEO_CON_RADAR_(SI APLICA)
++++++++++

Si el proyecto es grande:
- Subir RADAR_INDEX o RADAR_CORE segs.
- La IA usa RADAR como mapa, no “adivina” rutas.

==========
04.00_PROTOCOLO_DE_ENTREGAS_POR_TANDAS
==========

++++++++++
04.10_CUANDO_USAR_TANDAS
++++++++++

Usar tandas cuando:
- El archivo es demasiado largo.
- El chat limita tamaño.
- Se necesita evitar errores por “cortar por la mitad”.

++++++++++
04.20_FORMATO_OBLIGATORIO_TANDA
++++++++++

Encabezado:
- “TANDA X/Y”
- Archivo objetivo
- Acción: REEMPLAZAR / CONCATENAR / AGREGAR
- Nota: “No mezclar con otras tandas”

Ejemplo:
TANDA 2/3
ARCHIVO: 09.STANDAR.RADAR.txt
ACCION: CONCATENAR AL FINAL (orden estricto)

++++++++++
04.30_REGLA_DE_CIERRE
++++++++++

La última tanda debe incluir:
- “FIN_ARCHIVO” (marcador)
- checksum humano simple opcional:
  - líneas totales aproximadas, o
  - último título WBS esperado

Objetivo: evitar “creí que era el final”.

==========
05.00_PROTOCOLO_DE_REEMPLAZOS_(REPLACE)_Y_PARCHES
==========

05.10_REEMPLAZO_EXACTO
- Se usa cuando el usuario hará Ctrl+F “replace” en VS Code.
- La IA debe entregar:
  - TITULO/SUBTITULO objetivo
  - BLOQUE_ORIGINAL completo (tal como está)
  - BLOQUE_CORREGIDO completo

05.20_AGREGAR
- La IA debe:
  - indicar dónde insertar (título superior e inferior)
  - dar el bloque nuevo completo
  - no alterar nada fuera de ese bloque

05.30_ELIMINAR
- La IA debe:
  - mostrar exactamente el bloque a eliminar
  - explicar por qué
  - no borrar contexto conceptual salvo orden humana

==========
06.00_VALIDACION_DE_INTEGRIDAD_(ANTI-TRUNCACION)
==========

06.10_SEÑALES_DE_TRUNCACION
- Falta CHANGELOG.
- Cortes a mitad de título.
- Metadatos incompletos.
- “The file is too long and has been truncated” (si aplica).

06.20_REGLA_DURA
- Si hay truncación, no se emiten cambios canónicos.
- Se piden tandas adicionales o extractos.

06.30_EVIDENCIA
- Registrar en TROUBLE_LOG si truncación bloquea trabajo crítico.

==========
07.00_ERRORES_COMUNES_Y_MITIGACIONES
==========

E1) Mezclé versiones:
- Mitigar: IA siempre repite ID_UNICO + VERSION en su respuesta.

E2) Replace falló:
- Mitigar: bloque original debe ser exacto; si no, pedir extracto.

E3) Archivo equivocado:
- Mitigar: IA confirma “archivo y ruta lógica” antes de editar.

E4) Tandas desordenadas:
- Mitigar: formato TANDA X/Y + orden estricto + “FIN_ARCHIVO”.

function Remove-ExistingSegments {
  param([string]$ActivePath)

function Move-OldActiveFiles {
  param(
    [hashtable]$ActivePaths,
    [string]$OldRoot
  )
  $stamp = Get-NowStamp
  $dest = Join-Path $OldRoot $stamp
  Test-EnsureDirectory -Path $dest

    $p = $ActivePaths[$k]

      $name = Split-Path $p -Leaf
      Move-Item -LiteralPath $p -Destination (Join-Path $dest $name) -Force
    }
    Remove-ExistingSegments -ActivePath $p
  }
}

function Convert-ToRelativePath {
  param(
    [Parameter(Mandatory=$true)][string]$Root,
    [Parameter(Mandatory=$true)][string]$Full
  )
  $rootNorm = [System.IO.Path]::GetFullPath($Root).TrimEnd('\')
  $fullNorm = [System.IO.Path]::GetFullPath($Full)
  if ($fullNorm.StartsWith($rootNorm, [System.StringComparison]::OrdinalIgnoreCase)) {
    $rel = $fullNorm.Substring($rootNorm.Length)
    return $rel.TrimStart('\')
  }
  return $Full
}

function Get-Sha256Hex {
  param([Parameter(Mandatory=$true)][string]$Path)
  try {
    $h = Get-FileHash -Algorithm SHA256 -LiteralPath $Path
    return $h.Hash
  } catch {
    return "HASH_ERROR"
  }
}

function Split-IfOverSize {
  param(
    [Parameter(Mandatory=$true)][string]$ActivePath,
    [int] $MaxBytes = (8 * 1024 * 1024)
  )

  if (-not (Test-Path -LiteralPath $ActivePath)) { return }

  $len = (Get-Item -LiteralPath $ActivePath).Length
  if ($len -le $MaxBytes) { return }

  $prefix = Get-SegmentPrefix -ActivePath $ActivePath
  $lines = Get-Content -LiteralPath $ActivePath -Encoding UTF8

  $segIndex = 1
  $buffer = New-Object System.Collections.Generic.List[string]
  $bufferBytes = 0

  foreach ($line in $lines) {
    # Aproximación bytes UTF-8 para segmentación por líneas
    $lineBytes = [System.Text.Encoding]::UTF8.GetByteCount($line + "`n")
    if (($bufferBytes + $lineBytes) -gt $MaxBytes -and $buffer.Count -gt 0) {
      $segName = "{0}{1:000}" -f $prefix, $segIndex
      [System.IO.File]::WriteAllLines($segName, $buffer, [System.Text.Encoding]::UTF8)
      $segIndex++
      $buffer.Clear()
      $bufferBytes = 0
    }
    $buffer.Add($line)
    $bufferBytes += $lineBytes
  }

  if ($buffer.Count -gt 0) {
    $segName = "{0}{1:000}" -f $prefix, $segIndex
    [System.IO.File]::WriteAllLines($segName, $buffer, [System.Text.Encoding]::UTF8)
  }
}

      Modified  = $f.LastWriteTimeUtc.ToString("yyyy-MM-dd HH:mm:ss 'UTC'")
      Ext       = $extLower
      Type      = $logicalType
      Sha256    = $sha
    }) | Out-Null
  }

  # Orden determinista por Path (relativo)
  $recordsSorted = $records | Sort-Object -Property Path

  # ============================
  # 04.00_WRITE INDEX
  # ============================

  $ts = (Get-Date).ToString("yyyy-MM-dd HH:mm:ss")
  $indexLines = New-Object System.Collections.Generic.List[string]
  $indexLines.Add("==========") | Out-Null
  $indexLines.Add("RADAR_INDEX.ACTIVE") | Out-Null
  $indexLines.Add("==========") | Out-Null
  $indexLines.Add("") | Out-Null
  $indexLines.Add("TIMESTAMP.........: $ts (America/Santiago)") | Out-Null
  $indexLines.Add("ROOT..............: $rootFull") | Out-Null
  $indexLines.Add("PATH_MODE.........: $PathMode") | Out-Null
  $indexLines.Add("HASH_MODE.........: $HashMode") | Out-Null
  $indexLines.Add("TOTAL_FILES.......: $($recordsSorted.Count)") | Out-Null
  $indexLines.Add("") | Out-Null
  $indexLines.Add("----------") | Out-Null
  $indexLines.Add("FILES") | Out-Null
  $indexLines.Add("----------") | Out-Null

  foreach ($r in $recordsSorted) {
    $indexLines.Add("PATH..............: $($r.Path)") | Out-Null
    $indexLines.Add("SIZE_BYTES........: $($r.SizeBytes)") | Out-Null
    $indexLines.Add("MODIFIED..........: $($r.Modified)") | Out-Null
    $indexLines.Add("EXT...............: $($r.Ext)") | Out-Null
    $indexLines.Add("TIPO_LOGICO.......: $($r.Type)") | Out-Null
    if (-not [string]::IsNullOrWhiteSpace($r.Sha256)) {
      $indexLines.Add("SHA256............: $($r.Sha256)") | Out-Null
    } else {
      $indexLines.Add("SHA256............: ") | Out-Null
    }
    $indexLines.Add("") | Out-Null
  }

  [System.IO.File]::WriteAllLines($active.Index, $indexLines, [System.Text.Encoding]::UTF8)

  # ============================
  # 05.00_WRITE CORE
  # ============================

  $coreLines = New-Object System.Collections.Generic.List[string]
  $coreLines.Add("==========") | Out-Null
  $coreLines.Add("RADAR_CORE.ACTIVE") | Out-Null
  $coreLines.Add("==========") | Out-Null
  $coreLines.Add("") | Out-Null
  $coreLines.Add("TIMESTAMP.........: $ts (America/Santiago)") | Out-Null
  $coreLines.Add("ROOT..............: $rootFull") | Out-Null
  $coreLines.Add("MAX_CORE_BYTES.....: $MaxCoreFileBytes") | Out-Null
  $coreLines.Add("") | Out-Null

  foreach ($r in $recordsSorted) {
    $extLower = $r.Ext
    $isEligible = Test-IsCoreEligibleExt -ExtLower $extLower

    $coreLines.Add("----------") | Out-Null
    $coreLines.Add("FILE_BEGIN") | Out-Null
    $coreLines.Add("----------") | Out-Null
    $coreLines.Add("PATH..............: $($r.Path)") | Out-Null
    $coreLines.Add("SIZE_BYTES........: $($r.SizeBytes)") | Out-Null
    $coreLines.Add("MODIFIED..........: $($r.Modified)") | Out-Null
    $coreLines.Add("EXT...............: $($r.Ext)") | Out-Null
    $coreLines.Add("TIPO_LOGICO.......: $($r.Type)") | Out-Null

    if (-not $isEligible) {
      $coreLines.Add("CORE_STATUS.......: SKIPPED_NOT_TEXT") | Out-Null
      $coreLines.Add("") | Out-Null
      continue
    }

    if ($r.SizeBytes -gt $MaxCoreFileBytes) {
      $coreLines.Add("CORE_STATUS.......: SKIPPED_TOO_LARGE") | Out-Null
      $coreLines.Add("") | Out-Null
      continue
    }

    # Lectura segura
    try {
      $coreLines.Add("CORE_STATUS.......: INCLUDED") | Out-Null
      $coreLines.Add("++++++++++") | Out-Null
      $coreLines.Add("CONTENT") | Out-Null
      $coreLines.Add("++++++++++") | Out-Null

      $content = Get-Content -LiteralPath $r.FullPath -Raw -ErrorAction Stop
      $coreLines.Add($content) | Out-Null
      $coreLines.Add("") | Out-Null
    } catch {
      $coreLines.Add("CORE_STATUS.......: READ_ERROR") | Out-Null
      $coreLines.Add("ERROR.............: $($_.Exception.Message)") | Out-Null
      $coreLines.Add("") | Out-Null
    }
  }

  [System.IO.File]::WriteAllText($active.Core, ($coreLines -join "`n"), [System.Text.Encoding]::UTF8)

  # ============================
  # 06.00_WRITE FULL (INDEX + CORE + TREE_SIZE)
  # ============================

  # TREE_SIZE (bytes totales por carpeta) - determinista por path
  $dirSizes = @{}
  foreach ($r in $recordsSorted) {
    $dir = Split-Path $r.Path -Parent
    if ([string]::IsNullOrWhiteSpace($dir)) { $dir = "." }
    if (-not $dirSizes.ContainsKey($dir)) { $dirSizes[$dir] = 0L }
    $dirSizes[$dir] = $dirSizes[$dir] + [int64]$r.SizeBytes
  }

  $treeLines = New-Object System.Collections.Generic.List[string]
  $treeLines.Add("==========") | Out-Null
  $treeLines.Add("TREE_SIZE") | Out-Null
  $treeLines.Add("==========") | Out-Null
  $treeLines.Add("") | Out-Null

  foreach ($k in ($dirSizes.Keys | Sort-Object)) {
    $treeLines.Add("DIR...............: $k") | Out-Null
    $treeLines.Add("SIZE_BYTES........: $($dirSizes[$k])") | Out-Null
    $treeLines.Add("") | Out-Null
  }

  $fullLines = New-Object System.Collections.Generic.List[string]
  $fullLines.Add("==========") | Out-Null
  $fullLines.Add("RADAR_FULL.ACTIVE") | Out-Null
  $fullLines.Add("==========") | Out-Null
  $fullLines.Add("") | Out-Null
  $fullLines.Add("TIMESTAMP.........: $ts (America/Santiago)") | Out-Null
  $fullLines.Add("ROOT..............: $rootFull") | Out-Null
  $fullLines.Add("") | Out-Null

  $fullLines.Add("==========") | Out-Null
  $fullLines.Add("INCLUDE: RADAR_INDEX.ACTIVE") | Out-Null
  $fullLines.Add("==========") | Out-Null
  $fullLines.Add("") | Out-Null
  $fullLines.AddRange([System.IO.File]::ReadAllLines($active.Index, [System.Text.Encoding]::UTF8)) | Out-Null
  $fullLines.Add("") | Out-Null

  $fullLines.Add("==========") | Out-Null
  $fullLines.Add("INCLUDE: RADAR_CORE.ACTIVE") | Out-Null
  $fullLines.Add("==========") | Out-Null
  $fullLines.Add("") | Out-Null
  $fullLines.AddRange([System.IO.File]::ReadAllLines($active.Core, [System.Text.Encoding]::UTF8)) | Out-Null
  $fullLines.Add("") | Out-Null

  $fullLines.AddRange($treeLines) | Out-Null

  [System.IO.File]::WriteAllText($active.Full, ($fullLines -join "`n"), [System.Text.Encoding]::UTF8)

  # ============================
  # 07.00_WRITE LITE (diff básico vs último INDEX en old)
  # ============================

  # Buscar el INDEX más reciente en OldDir (si existe) para diff de paths
  $prevIndex = $null
  $oldIndexCandidates = Get-ChildItem -LiteralPath $OldDir -Recurse -File -ErrorAction SilentlyContinue |
    Where-Object { $_.Name -eq "RADAR_INDEX.ACTIVE.txt" } |
    Sort-Object -Property LastWriteTimeUtc -Descending

  if ($oldIndexCandidates -and $oldIndexCandidates.Count -gt 0) {
    $prevIndex = $oldIndexCandidates[0].FullName
  }

  $currentPaths = $recordsSorted | ForEach-Object { $_.Path }

  $prevPaths = @()
  if ($prevIndex) {
    try {
      # Parse simple: extraer líneas PATH..............:
      $prevLines = Get-Content -LiteralPath $prevIndex -Encoding UTF8 -ErrorAction Stop
      $prevPaths = $prevLines |
        Where-Object { $_ -like "PATH..............:*" } |
        ForEach-Object { ($_ -split "PATH..............:\s*",2)[1].Trim() } |
        Where-Object { -not [string]::IsNullOrWhiteSpace($_) }
    } catch {
      $prevPaths = @()
    }
  }

  $setPrev = New-Object "System.Collections.Generic.HashSet[string]" ([string[]]$prevPaths)
  $setCurr = New-Object "System.Collections.Generic.HashSet[string]" ([string[]]$currentPaths)

  $newFiles = New-Object System.Collections.Generic.List[string]
  foreach ($p in $setCurr) { if (-not $setPrev.Contains($p)) { $newFiles.Add($p) | Out-Null } }
  $deletedFiles = New-Object System.Collections.Generic.List[string]
  foreach ($p in $setPrev) { if (-not $setCurr.Contains($p)) { $deletedFiles.Add($p) | Out-Null } }

  # Modified basado en metadata (size+modified) comparando contra prevIndex es complejo sin parse completo.
  # MVP: marcar MODIFIED como vacío y declarar limitación.
  $modifiedFiles = @()

  $liteLines = New-Object System.Collections.Generic.List[string]
  $liteLines.Add("==========") | Out-Null
  $liteLines.Add("RADAR_LITE.ACTIVE") | Out-Null
  $liteLines.Add("==========") | Out-Null
  $liteLines.Add("") | Out-Null
  $liteLines.Add("TIMESTAMP.........: $ts (America/Santiago)") | Out-Null
  $liteLines.Add("ROOT..............: $rootFull") | Out-Null
  $liteLines.Add("TOTAL_FILES.......: $($recordsSorted.Count)") | Out-Null
  $liteLines.Add("DIFF_BASE.........: $([string]::IsNullOrWhiteSpace($prevIndex) ? 'NONE' : $prevIndex)") | Out-Null
  $liteLines.Add("") | Out-Null
  $liteLines.Add("NOTA..............: MODIFIED_FILES (MVP) no se calcula en esta versión; solo NEW/DELETED por paths.") | Out-Null
  $liteLines.Add("POINTER_INDEX......: $($active.Index)") | Out-Null
  $liteLines.Add("POINTER_CORE.......: $($active.Core)") | Out-Null
  $liteLines.Add("POINTER_FULL.......: $($active.Full)") | Out-Null
  $liteLines.Add("") | Out-Null

  $liteLines.Add("----------") | Out-Null
  $liteLines.Add("NEW_FILES") | Out-Null
  $liteLines.Add("----------") | Out-Null
  foreach ($p in ($newFiles | Sort-Object)) { $liteLines.Add($p) | Out-Null }
  $liteLines.Add("") | Out-Null

  $liteLines.Add("----------") | Out-Null
  $liteLines.Add("MODIFIED_FILES") | Out-Null
  $liteLines.Add("----------") | Out-Null
  foreach ($p in $modifiedFiles) { $liteLines.Add($p) | Out-Null }
  $liteLines.Add("") | Out-Null

  $liteLines.Add("----------") | Out-Null
  $liteLines.Add("DELETED_FILES") | Out-Null
  $liteLines.Add("----------") | Out-Null
  foreach ($p in ($deletedFiles | Sort-Object)) { $liteLines.Add($p) | Out-Null }
  $liteLines.Add("") | Out-Null

  [System.IO.File]::WriteAllLines($active.Lite, $liteLines, [System.Text.Encoding]::UTF8)

  # ============================
  # 08.00_SEGMENTACION 8MB
  # ============================

  foreach ($p in @($active.Lite, $active.Index, $active.Core, $active.Full)) {
    Split-IfOverSize -ActivePath $p
  }

  # ============================
  # 09.00_FINAL VALIDATION (OK/FAIL real)
  # ============================

  if (-not (Test-ValidateOutputsExist -ActivePaths $active)) {
    Write-Error "FAIL: Outputs activos faltantes o vacíos."
    exit 1
  }

  Write-Output "OK: RADAR generado correctamente."
  Write-Output "OUTPUTS:"
  Write-Output " - $($active.Lite)"
  Write-Output " - $($active.Index)"
  Write-Output " - $($active.Core)"
  Write-Output " - $($active.Full)"
  exit 0

} catch {
  Write-Error ("FAIL: " + $_.Exception.Message)
  exit 1
}
