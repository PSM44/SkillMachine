01.00_PROPOSITO

01.10_EVITAR_CAOS_DE_ARCHIVOS_SUELTOS

Sin reglas, cada archivo arrastrado:

- Puede quedar sin contexto.
- Puede duplicar versiones viejas.
- Puede romper consistencia de nombres y estándares.

Este skill define un pipeline mínimo para evitar eso.

01.20_CUANDO_APLICA

Siempre que:

- Se arrastra un archivo a la IA para:
  - Revisar.
  - Documentar.
  - Integrar a un sistema.
  - Depurar un fallo.

==========
02.00_PIPELINE_DRAG_AND_DROP

02.10_PASO_0:_STAGING_LOGICO

Por defecto, todo archivo drag&drop está en un estado de STAGING lógico:

- Aún no es canon.
- Aún no forma parte del ecosistema PF/EVA/WINGS.
- Sirve como insumo para análisis, refactor o diseño.

La ruta física concreta de STAGING se define en la taxonomía/estructura de proyecto.

02.20_PASO_1:_IDENTIFICAR_TIPO_Y_PROPOSITO

Al recibir un archivo, la IA debe:

- Detectar tipo (txt, ps1, py, xlsx, pdf, etc.).
- Confirmar propósito:
  - ¿Revisión?
  - ¿Migración a un sistema?
  - ¿Refactor de contenido?
  - ¿Depuración de un fallo (TROUBLE/INCIDENT)?

Sin propósito claro, hay alto riesgo de trabajo inútil.

02.30_PASO_2:_MAPEO_A_SISTEMA_Y_UBICACION

Antes de proponer cambios profundos, la IA debe:

- Preguntar (o inferir si lo dijiste) a qué sistema/proyecto pertenece el archivo.
- Sugerir dónde podría vivir en el futuro (carpeta lógica, ID_FS aproximado).
- Dejar claro que la decisión final es tuya.

02.40_PASO_3:_ANALISIS_Y_PROPUESTAS

Según tipo y propósito:

- Aplicar skills adecuados:
  - FILE_CONTENT para estructura interna.
  - HUMAN_README si se trata de documentación humana.
  - CODE/DEV_ENV si es script.
- Proponer:
  - Mejoras de estructura y comentarios.
  - División si es demasiado grande.
  - Normalización de nombres conforme WBS_ID.

02.50_PASO_4:_INTEGRACION_O_DESCARTE

Decisión posible:

- Integrar:
  - Se asigna ID_FS según WBS_ID.
  - Se define ruta destino en el ecosistema.
  - Se actualiza HUMAN/BATON/RADAR si corresponde.

- Mantener solo como referencia:
  - Se marca explícitamente como “no canon”.
  - Puede guardarse en carpeta de evidencia/histórico.

- Descartar:
  - Solo si se decide explícitamente que no aporta valor.

==========
03.00_VERSIONES_Y_CANON

03.10_ARCHIVO_DRAG_DROP_VS_ARCHIVO_CANON

Si existe ya un archivo “canon” en el ecosistema:

- El archivo drag&drop no debe asumir que lo reemplaza.
- Debe tratarse inicialmente como:
  - Versión vieja.
  - Variante alternativa.
  - Evidencia de un estado anterior.

03.20_REGISTRO_DE_NO_CANON

Si decides conservar el archivo drag&drop como NO CANON:

- Debe quedar claro en:
  - Su nombre (ej. sufijo `_NONCANON` o carpeta específica).
  - HUMAN/README o TROUBLE/INCIDENT que lo referencie.

Nunca debe pisar un archivo canon sin análisis explícito.

==========
04.00_RELACION_CON_TROUBLE_INCIDENTS

04.10_DRAG_AND_DROP_PARA_DEBUG

Si arrastras un archivo porque hay un fallo:

- Es muy probable que debas crear al menos un TROUBLE.
- Se debe registrar:
  - Archivo recibido.
  - Contexto del fallo.
  - Hipótesis inicial de causa.

04.20_DRAG_AND_DROP_TRAS_INCIDENTE

Si el archivo arrastrado está vinculado a un INCIDENT:

- Debe quedar referido en el INCIDENT:
  - Como evidencia.
  - Como objeto de refactor o reescritura.

Al cierre del INCIDENT:
- Se debe decidir si:
  - El archivo se integra como nueva versión canon.
  - Se archiva como histórico.
  - Se descarta.

==========
05.00_INTERACCION_CON_IA

05.10_QUE_PUEDE_PROPONER_LA_IA

La IA puede:

- Proponer ID_FS para el futuro archivo integrado.
- Proponer estructura interna según FILE_CONTENT.
- Extraer HUMAN.README desde un PDF/Word caótico.
- Generar scripts auxiliares para renombrar/mover archivos.

05.20_LIMITES

La IA no debe:

- Asumir que el archivo drag&drop es canon.
- Borrar versiones previas sin instrucción explícita.
- Mezclar dominios (PF, EVA, WINGS) si no se le indica.
